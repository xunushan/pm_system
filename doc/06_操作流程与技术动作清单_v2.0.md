# 一、操作流程

## A. Story 1：目标设立与拆解（原 Story 0+1）

| 步骤 | 用户动作 | 系统行为 |
|------|----------|----------|
| 1 | 发送 `/pm 帮我规划...` | Hermes Agent → 主 Skill `pm` → 解析"规划" → `skill_view("pm-plan")` |
| 2 | pm-plan 提问引导（含目标时间范围、scheduled_start_date） | pm-plan 按引导策略提问（专题构成→目标时间范围→逐专题阶段任务），每专题确认后进下一个。逐专题生成内容追加到同一个 draft（调 Service 写 drafts） |
| 3 | 用户聊天回复 | 讨论态，pm-plan LLM 理解意图调整规划结构（调 Service 更新 drafts） |
| 4 | 所有专题讨论完 | pm-plan 发总览卡片（只展示概览：专题数/阶段数/任务数，不展示全部细节）+ 确认方案按钮（按钮 action_value 只含 draft_id） |
| 5 | 用户点击 **story1_确认方案** | 飞书回调（只传 draft_id，规避约 30KB 数据量限制）→ Service `/webhook/feishu/card` → 硬编码路由 → 方案确认服务：用 draft_id 读 drafts → 事务写入 goals/themes/phases/tasks（正式态，含目标时间范围+scheduled_start_date，无 deadline/executor）→ 删 drafts → 飞书推送"方案已保存，H5 链接：..." |
| 6 | 用户在 H5 页面 CRUD | H5 页面调 Service API → 落库（详见 Story 9） |

> **要点**：pm-plan 全程聊天沟通，逐专题确认，内容追加到 drafts。总览卡片只展示概览，确认按钮只传 draft_id（规避回调数据量限制）。drafts 纯存储，不同步展示。规划态产出：目标（名称+时间范围+scheduled_start_date+描述）、专题（名称+类型+排序）、阶段（名称+排序）、任务（名称）。无 deadline、executor、is_draft。

## B. Story 2：首次纳入调度（含项目空间设置，合入原 Story 8）

| 步骤 | 用户动作 | 系统行为 |
|------|----------|----------|
| 1 | 发送 `/pm 开始调度` 或 H5 页面点"激活" | Hermes Agent → 主 Skill `pm` → 生成卡片 A（选专题+设项目空间） |
| 2 | 用户多选专题 + 设各专题 managed/path | 卡片 A：多选专题（专题无序并列）+ 每个专题 managed 状态（默认托管，可改关联+填 path，已关联显示当前 path）→ 点"下一步" |
| 3 | 飞书回调 → Service patch 卡片 A 为卡片 B | 查每个选中专题 sort_order 最小的未开始阶段（自动锁定）→ 校验全局进行中+本次≤3 → managed=0 的校验 path 存在性（不存在不允许）→ 卡片 B（各阶段 + deadline date_picker + 确认调度按钮） |
| 4 | 用户填各 deadline，点 **story2_确认调度** | 飞书回调 → 硬编码路由 → 调度确认服务：① 事务 UPDATE phases（进行中, activated_at=NOW(), deadline）+ 即时级联 themes/goals + status_change_log（forward）；② 异步工作空间初始化（managed 分支：managed=1→mkdir+git init+骨架含规范文件；managed=0→校验 path，不创建任何文件）→ 飞书推送"调度已确认" |

> **要点**：patch 卡片形式；多选专题；阶段自动锁定（强约束）；managed/path 在卡片 A 设置（激活前最后一刻）；managed=0 不创建任何文件；deadline 必填；不碰 scheduled_start_date；首次与追加统一；激活后不能改 managed；记录 phases.activated_at。

## C. Story 3：当日计划推送

| 步骤 | 用户动作 | 系统行为 |
|------|----------|----------|
| 1 | 定时 8:30 / `/pm 今日计划` / 10:00 巡检提醒后 | Hermes Agent → 主 Skill `pm` → `skill_view("pm-daily")` |
| 2 | 任务池预查询（Service 代码） | 已激活阶段（activated_at 有值）下的待执行任务 + 排除已暂停 |
| 3 | 今日计划决策（LLM） | pm-daily 综合昨日完成+进度+deadline+依赖 → 候选任务 5-8 个 |
| 4 | executor 推断（LLM） | pm-daily 按专题 type 推断（learning/research/source→人；dev/survey→智能体） |
| 5 | 前置子任务生成（LLM） | pm-daily 调 pm-subtask 生成"今日整体前置"（不按单个任务，与任务解耦）。只对 executor=人 的任务生成 |
| 6 | 生成可交互卡片 | 候选任务勾选框（每行：任务名（阶段名(deadline)）[executor]）+ 前置勾选框（两组独立）+ 确认今日计划按钮（确认前数据只在卡片） |
| 7 | 用户勾选任务 3-5 个 + 勾选前置（可取消） | 卡片勾选交互，两组独立 |
| 8 | 用户点击 **story3_确认今日计划** | 飞书回调 → 硬编码路由 → 每日计划确认服务：轻量事务 ① INSERT daily_records（已确认）；② INSERT daily_tasks（勾选的）；③ INSERT subtasks（勾选的前置，待执行）→ 立即返回飞书 200（<200ms） |
| 8.5 | （异步触发） | ① opencode run 执行勾选的前置任务（智能体执行）；② 若勾选任务有 executor=智能体 → 启动 opencode serve（动态端口） |

> **要点**：今日计划走卡片；LLM 决策带候选供勾选；executor 按专题 type 推断只读展示；前置按今日整体生成、与任务解耦、两组独立勾选、可取消；前置只对人执行任务；无 drafts；不确认只提醒。

## D. Story 4A：工作空间任务执行（智能体）

| 步骤 | 用户动作 | 系统行为 |
|------|----------|----------|
| 1 | （承接 Story 3 步骤 7.5） | 异步 HTTP POST 到 OpenCode，携带任务清单（按项目空间规范 claude.md 类产出文件） |
| 2 | 系统记录超时监控 | SET task_timeout:{task_id} EX 7200（Redis） |
| 3 | OpenCode 执行 | 写产出文件 |
| 4 | OpenCode 回调 | 调用产出回调服务 |
| 5 | 系统处理回调 | 记录 workspace_progress → DEL task_timeout |
| 6 | Service 发验收卡片 + 发送产出文件 | Service 调飞书 API（模板填充，无 LLM）：验收卡片（任务名 + 产出文件名列表 + 验收通过/需要修改按钮）+ 逐个发送产出文件到飞书 |
| 7 | 用户点击 **story4A_验收通过** | 飞书回调 → 硬编码路由 → 产出验收服务：事务 ① UPDATE tasks 已完成；② UPDATE subtasks（相关前置）；③ 写 status_change_log；④ 即时级联 + 发阶段完成事件 → Supervisor 衔接；⑤ COMMIT |
| 8 | 用户点击 **story4A_需要修改** | 飞书回调 + feedback（卡片输入框直接提交）-> 读 retry_count：<3 则 retry_count+=1，重新 POST OpenCode + 重设 Redis；>=3 则退该 task 的 session（DELETE /session/:sessionID，全局 serve 保留）+ 飞书通知（工作空间路径 + session_id，提示用户本地接管，无按钮，确认完成走 /pm 聊天指令） |
| 9 | 用户手动处理完，发 `/pm 确认完成 [任务]` | 带任务名：Hermes Agent -> 主 Skill `pm` -> 直接调 TaskAppSvc.confirm_complete（标记完成+即时级联，无后置）；不带任务名：Skill 调 Service **推确认完成卡片**（D26）--该工作空间下已完成任务（只读+执行主体）+ 待确认任务（checker 勾选+执行主体），用户勾选确认完成；智能体任务可选"改交智能体重新执行"（改 executor=agent + 重新下发，与确认完成互斥）。回调 -> TaskAppSvc.confirm_complete -> 系统重新建/复用 session 驱动后续任务 |
| 10 | Redis 超时兜底 | 2h 未回调 → KeyExpirationEvent → 飞书通知"智能体执行超时" |

> **要点**：智能体在 Story 3 确认后唤起；产出统一为文件；验收卡片+发送文件到飞书；3 次不通过用户手动接管；"/pm 确认完成"后系统重新接管后续智能体任务；不生成前置/后置。

## E. Story 4B：工作空间任务执行（人）

| 步骤 | 用户动作 | 系统行为 |
|------|----------|----------|
| 1 | 用户完成后发送 `/pm 完成 [任务名]` | Hermes Agent → 主 Skill `pm` → `skill_view("pm-subtask")` |
| 2 | 系统定位任务 | 调任务查询服务；唯一匹配直接进入④，多匹配进③ |
| 3 | 歧义澄清（可选） | 候选列表，用户聊天确认 |
| 4 | 标记完成 + 生成后置（pm-subtask Skill 执行） | pm-subtask：① 调 TaskAppSvc 标记完成 + 即时级联（任务此时已完成，即时级联触发）；② pm-subtask（type=后置）基于模板+工作空间文件快照 → LLM 生成后置清单（只对人执行任务） |
| 5 | 生成后置勾选卡片 | 已完成任务 + 后置清单（每项勾选框，默认全选）+ 确认后置按钮 + 不需要后置按钮 |
| 6 | 用户卡片勾选（取消不想要的，可全取消，不能新增） | 卡片勾选交互 |
| 7 | 用户点击 **story4B_确认后置** 或 **story4B_不需要后置** | 飞书回调 → 硬编码路由 → 后置确认服务：INSERT subtasks（勾选的后置，待执行；全取消则不插入）。事务后异步：opencode run 执行后置（智能体执行） |
| 8 | 后置执行完成 | opencode run 退出 → 子任务状态更新 → 飞书非阻塞通知（失败也非阻塞） |

> **要点**：完成走指令"/pm 完成"；pm-subtask 标记完成（即时级联）+ 生成后置，完成和后置脱钩；后置只对人执行任务；后置卡片勾选可全取消；后置智能体执行；不保留"需人工介入"。

## F. Story 5：日终总结

| 步骤 | 用户动作 | 系统行为 |
|------|----------|----------|
| 1 | 用户 `/pm 今日总结` 或 21:00 定时 | Hermes Agent → 主 Skill `pm` → `skill_view("pm-summary")` |
| 2 | 统计预查询（Service 代码） | 调日统计服务 → 今日完成/未完成/阶段健康度（状态已由即时级联更新） |
| 3 | 文案与建议生成（LLM） | pm-summary → 总结文案 + 建议 |
| 4 | 推送总结卡片 | 今日任务列表（每项带状态切换按钮：未完成显示[标记完成]，已完成显示[标记未完成]）+ 步骤进展 + 建议 + 确认日终总结按钮 |
| 5 | 用户点状态切换按钮（可选，双向） | 飞书回调（action_value 含 task_id）→ Service PATCH 任务状态 → 即时级联 → 重新统计 → 用 message_id 刷新原卡片（按钮文案反转） |
| 6 | 用户点击 **story5_确认日终总结** | 飞书回调 → 硬编码路由 → 日终总结确认服务：UPDATE daily_records SET is_confirmed=1 → 异步写 daily.md。不级联 |

> **要点**：纯回顾+快照+建议；异议走卡片直接改状态（按钮动作形态，双向，直接改不弹确认）；改状态触发即时级联；不确认不强制。

## G. Story 6：周总结

| 步骤 | 用户动作 | 系统行为 |
|------|----------|----------|
| 1 | 周日 12:00 定时 | Hermes Cron → 主 Skill `pm` → `skill_view("pm-summary")` |
| 2 | 统计预查询（Service 代码） | 调周统计服务 |
| 3 | 文案与下周建议生成（LLM） | pm-summary → 周总结文案 + 下周建议（参考 Supervisor 衔接状态） |
| 4 | 推送总结卡片 | 已阅按钮 |
| 5 | 用户聊天异议（可选） | LLM 理解 → 重新预查询（不修改底层状态）→ 刷新卡片。修正回当天日终或 H5 页面 |
| 6 | 用户点击 **story6_已阅周总结** | 飞书回调 → 硬编码路由 → 周总结确认服务：INSERT/UPDATE weekly_records → 异步写 weekly.md |

## H. Story 7：子任务配置

| 步骤 | 用户动作 | 系统行为 |
|------|----------|----------|
| 1 | 用户在 H5 页面"子任务模板"板块 CRUD | H5 页面调 Service API（POST/PUT/DELETE /api/v1/subtask-templates）→ 落库（代码，不建 Skill）。页面形态 B：列表直接展示带所属，新增时表单选所属 |
| 2 | 配置保存 | 立即生效，下次生成计划时自动应用（只对 executor=人 的任务应用，阶段级优先于专题级，同名去重） |

> **要点**：配置走 H5 页面形态 B；配置粒度专题/阶段；不校验专题 type；去掉文件上传/下载。

## I. Story 8：主动巡检与阶段衔接（原 Story 9）

| 步骤 | 触发 | 系统行为 |
|------|------|----------|
| 1 | 阶段完成事件 | Supervisor 查同专题 sort_order+1 下一阶段 → 推衔接卡片（带 deadline date_picker + 确认激活/暂不激活） |
| 2 | 用户点击 **story8_确认激活** | 飞书回调 → 硬编码路由 → 调度激活服务 → UPDATE phases（进行中, activated_at）+ 即时级联 + 异步工作空间初始化（managed 分支） |
| 3 | 专题完成 | Supervisor 推卡片列出未完成的其他专题（单选）→ 用户选后 patch 为填 deadline → 确认激活 |
| 4 | scheduled_start_date 到了未激活 | Supervisor 每天巡检 → 推提醒"你计划今天开始，要激活吗"→ 跳 Story 2 激活。激活后停止 |
| 5 | deadline 临近（前1天/当天） | Supervisor 每天巡检 → 推进度提醒 + 跳 H5 页面按钮 |
| 6 | 未确认计划（10:00） | Supervisor → 推提醒 |
| 7 | 未做日终总结（21:00） | Supervisor → 推提醒 |
| 8 | 阶段衔接 24h 未响应 | Supervisor → 再推一次衔接提醒 |

> **要点**：阶段衔接带 deadline 填写；专题完成给专题清单选（专题无序）；新增 scheduled_start_date 提醒；卡顿监测暂不做；Supervisor 纯代码。

## J. Story 9：轻量编辑与状态回退（原 Story 10）

| 步骤 | 用户动作 | 系统行为 |
|------|----------|----------|
| 1 | H5 页面字段编辑（名称/描述/deadline/executor） | H5 页面调 Service API → 落库 |
| 2 | 增删任务（已激活阶段下，物理删除） | H5 页面调 Service API → 落库 |
| 3 | 阶段排序（拖拽） | H5 页面调 Service API → 落库（任务排序不支持） |
| 4 | 状态变更：暂停（进行中→已暂停） | H5 页面填 reason → Service 校验状态机 → UPDATE status + status_change_log（pause）→ 不纳入计划/不占名额 |
| 5 | 状态变更：恢复（已暂停→进行中） | H5 页面（不填 reason）→ Service → UPDATE status + status_change_log（resume）→ 重新纳入计划 |
| 6 | 状态变更：回退（已完成→待执行） | H5 页面填 reason → Service → UPDATE status + status_change_log（revert）→ 即时重算级联 |
| 7 | 查看项目空间状态（只读） | 专题详情显示 managed/path，激活后不能改 managed |

> **要点**：阶段排序保留、任务排序不支持；executor 可改；任务物理删除；managed/path 只读查看（设置在 Story 2）；暂停/回退填 reason、恢复不填；无卡顿监测字段。

## 表 2：卡片按钮点击（入口 B）

| 按钮文案 | 动作类别 | 对应 Story | 接收端点 | 硬编码路由 |

# 三、技术关键决策

> **状态唯一真相源** — 任务状态以 tasks 表为准，daily_tasks 为快照。

> **即时级联** — 任务/阶段状态变更时即时向上级联（事务内），不再依赖日终总结。回退同样触发即时重算级联。

> **H5 页面承载调整** — 规划确认后给 H5 链接，所有 CRUD、deadline、状态变更在页面。废弃多维表格、is_draft。drafts 表保留用于确认前存储规划数据（纯存储，不同步展示），规避飞书回调约 30KB 数据量限制（确认按钮只传 draft_id）。载体为简单 H5 页面（个人使用，无鉴权），支持 URL 参数预填。

> **pm 主路由纯路由+预填链接** — pm 不直接调 Service。生成类路由到子 Skill（子 Skill 调 Service）；配置类 LLM 提取参数 → 生成预填 H5 链接。

> **专题无序 + 阶段强约束** — 专题去 sort_order（并列无序），阶段是 roadmap（强约束，按 sort_order 顺序激活，自动锁定第1个未开始阶段）。

> **executor 自动推断** — 规划态不定 executor，pm-daily 按专题 type 推断（learning/research/source→人；dev/survey→智能体），卡片只读，要改走页面。

> **时间字段精简** — 去掉专题时间，保留目标时间范围（规划态确认）+ scheduled_start_date（规划态确认，提醒激活）+ 阶段 deadline（激活时填）。

> **前置/后置只服务人执行任务** — 智能体任务不生成前置/后置。

> **确认按钮设计** — 确认类操作仅执行轻量数据库写 + 即时级联（<200ms），耗时操作异步。

> **飞书回调 3 秒超时** — 确认类 API 仅执行数据库写 + 即时级联（纯 DB），立即返回。

> **数据库事务与 IO 分离** — 事务内禁止 IO/HTTP。

> **卡片刷新使用 message_id** — 更新消息接口刷新原卡片。

> **去掉"需人工介入"终态（4A）** — 3 次不通过不改状态，用户介入后"/pm 确认完成"，系统重新接管后续智能体任务（不同端口）。

> **两个完成指令** — 4B 用"/pm 完成"（带后置），4A 人工确认用"/pm 确认完成"（不带后置）。

> **验收发文件** — 验收卡片 + 发送产出文件到飞书，用文件名体现产出。

> **Story 5 异议卡片直接改状态** — 按钮动作形态，双向，直接改不弹确认。

> **Story 2 多选专题 + 合入项目空间设置** — 多选专题，每个激活第1个未开始阶段；managed/path 在卡片 A 设置（原 Story 8 并入）；激活后不能改 managed。

> **状态机约束** — 任务状态：待执行/已完成/已暂停（去掉需人工介入）。阶段/专题/目标支持已暂停 + 有限回退（必填 reason）。所有状态变更写 status_change_log。

> **工作空间初始化幂等性** — managed=1 初始化（mkdir+git init+骨架含规范文件）；managed=0 校验 path，不创建任何文件。激活后不改 managed。

> **子任务模板合并规则** — 阶段级优先于专题级，同名去重。

> **配置类操作不建 Skill** — 模板配置走 H5 页面 CRUD。

> **项目空间主 Agent 进程管理** — 首次下发智能体任务时启动（Story 3 确认后），阶段级常驻，端口动态分配，心跳每 5 分钟。不受 managed 影响。

> **Skill 与 Service 能力边界** — 仅 LLM 核心逻辑做 Skill（5 个），纯确定性逻辑为 Service 代码。

> **事件即时 + 定时巡检兜底** — 状态变更事件即时触发衔接/级联；Supervisor 定时巡检兜底。所有调整操作跳 H5 页面。

> **任务排序不页面定** — 交给 pm-daily 动态定；阶段排序保留。

> **任务物理删除** — 任务不要了物理删除（区别于模板标记 inactive）。

> **卡顿监测暂不做**。

> **重新编号** — Story 1（原0+1）、Story 8（原9）、Story 9（原10），原 Story 8 删除并入 Story 2。
