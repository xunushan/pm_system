## 目的

从 pm-system 复盘沉淀。目标：

1. 任务/变更/bug 并存时有确定性排序，不靠记忆。
2. 新 session 接手成本低（少 token 恢复进展）。
3. 子 agent 拿到的任务自包含、无上下文可接手。
4. 项目无关：拆分粒度自由（不绑 Story），后续项目通用。

## 一、三层分工

| 层 | 承载 | 性质 | 读取模型 |
|---|---|---|---|
| **progress.md** | 当前焦点 + 下一步 issue 指针 + 阻塞 | 纯指针，<20 行，可重写 | 新 session 接手先读此（聚合有序视图） |
| **issues** | 所有任务/变更/bug/spike 的详情 | 自包含，含验收+依赖+复用件 | 按标签过滤，子 agent 读详情 |
| **git** | 历史真相源 | commit + PR | `git log` 查做了什么 |

**原则**：progress.md 只存指针、issues 存详情、git 存历史，三者不重复（单一写入）。

### 为何保留 progress.md（不纯 issues）

四场景 token 推演：

| 场景 | 有 progress.md | 纯 issues |
|---|---|---|
| 派发+更新状态 | +1 次小 Edit | 省这次 |
| **新 session 接手** | 读 <20 行聚合视图，立即知焦点 | 读 N 个 open issue body 综合推断焦点 |
| 发现 bug 提交 | 无差异（都不进 progress） | 无差异 |
| 变更后加任务 | +1 行指针 | 省这行 |

只有"接手"场景 progress.md 价值显著（聚合 vs 离散综合），且接手是高频场景。其余三场景纯 issues 仅省几次小 Edit，量小。故保留极简 progress.md 作"有序指针层"。

> GitHub issue 无原生顺序（靠 milestone/priority 分组，脆弱），progress.md「下一步 1/2/3」天然有序，这是 issues 难以原生承载的独立价值。

### 极简 progress.md 模板

```markdown
# PROGRESS
> git + gh 是真相源；本文件只放"当前焦点+下一步指针"，详情看 issue。

## 当前焦点
<1-2 行：正在做什么、为什么，指向当前工作块>

## 下一步（≤3，按优先级）
1. issue#N - 一句话  验收：<…>
2. issue#M - 一句话  验收：<…>

## 阻塞 / 待决
- issue#K / 决策点 - 谁阻塞谁
```

粒度无关：焦点可是 Story / milestone / 临时工作块，progress.md 不关心粒度只存指针。

## 二、标签体系（3 维，兼容 triage）

采纳 triage 的 5 状态机 + 扩展类型层 + 优先级。每 issue **恰好一个状态 + 一个类型**，优先级可选。

### 状态（照搬 triage）

| 标签 | 含义 |
|---|---|
| `needs-triage` | 需评估（单人场景少用，处理外部/口述输入） |
| `needs-info` | 等待补充信息 |
| `ready-for-agent` | 自包含、依赖满足、AFK 可做（**核心，查下一任务用**） |
| `ready-for-human` | 需人工（判断/外部访问/设计决策/手测） |
| `wontfix` | 不做（已实现/拒绝/out-of-scope） |

### 类型（扩展 triage 的 bug/enhancement）

| 标签 | 用途 |
|---|---|
| `type:bug` | 功能损坏 |
| `type:feature` | 新功能/任务 |
| `type:spike` | 原型验证（§4 prototype） |
| `type:chore` | 杂项/重构 |
| `type:docs` | 文档 |

> **不含 `type:change`**：change 是流程（doc/10 §四：和用户讨论 -> 冲突检测 -> 生成 feature/bug/spike issue），不是 issue 类型。change 的产物是普通 issue，不单列类型。

### 优先级

`P0` / `P1` / `P2` / `P3`（与 reviewer 严重度对齐）。

### 不用 Story 标签

后续项目不一定按 Story 拆。归属用 issue 描述引用（"关联 doc/01 Story N"或"关联模块 X"）或 milestone 承载，不强制 `story-N` 标签。本项目历史 issue 标题 `[Story3]` 是约定，保留但不作标签体系核心。

## 三、issue 模板（GitHub form，非 skill）

用 `.github/ISSUE_TEMPLATE/*.yml`（form 带字段），让 issue 自包含。借鉴 [to-issues](file:///Users/isuntaiyang/projects/soft_engineering/.claude/skills/to-issues/SKILL.md) 模板（Parent/What to build/Acceptance/Blocked by）。

> **issue 只放任务自身信息，不放项目绑定字段**（铁律/复用件/模型选型）。项目绑定信息归 CLAUDE.md（子 agent 读项目 CLAUDE.md 获取，见 code-reviewer.md 实证），单一写入，issue 跨项目复用时不带噪音。

### 通用字段（所有类型）

- **What to build**：端到端行为描述（非逐层实现）。避免文件路径/代码片段（易过期，to-issues 原则）。例外：prototype 产出的状态机/契约代码比文字精确，可放上下文指针。
- **Acceptance criteria**：Given/When/Then 清单（任务自身的验收，该放）。
- **Blocked by**：`#N` 依赖（GitHub 原生渲染）；或"None - can start immediately"。
- **关联决策**：影响本任务的 D{N}（doc/07）。

### 各类型专属字段

- **bug**（`type:bug`）：复现步骤 + 期望 + 实际 + 严重度。triage 要 reproduce 验证。
- **spike**（`type:spike`）：要验证的问题 + 方法 + 期望产出（FACT 入 doc/07）+ `unblocks #N`。
- **feature/task**（`type:feature`）：通用字段。
- **chore/docs**：简化版通用字段。

> **模板创建方式**：项目初始化时在 `.github/ISSUE_TEMPLATE/` 创建 form（非 skill，非每次判断）。CLAUDE.md 补一句选型规范。

### 契约单 = issue + CLAUDE.md（不重复项目绑定字段）

子 agent 接手任务时读两份：
- **issue body**（任务自身信息）：What/Acceptance/Blocked by/关联 D{N}。
- **项目 CLAUDE.md**（项目环境）：铁律、复用件清单、状态机、范本路径。子 agent 会读 CLAUDE.md（实证）。

**不进 issue 的项目绑定字段**：
- 铁律 -> CLAUDE.md constitution
- 复用件清单 -> CLAUDE.md §十一（复用件清单）
- 范本路径 -> CLAUDE.md（model 范本等）
- **模型选型**（haiku/sonnet）-> 不进 issue、不进 CLAUDE.md，**派发时主 agent 据任务性质判断**（机械->haiku，业务->sonnet）。这是调度决策非任务固有属性。

单一写入：铁律只在 CLAUDE.md，issue 不重复。

## 四、任务/变更/bug 并存排序

三者性质差异决定排序：

| 类型 | 紧迫性 | 能否预排期 |
|---|---|---|
| bug | 高（已影响功能） | 需尽快，常打断 |
| 变更 | 中（设计层，影响多） | 走 doc/10 §四 |
| 任务 | 低（计划内） | 可排期 |

### 排序原则

1. **P0/P1 bug 优先于一切**：不修持续出错/阻塞。证据：本项目 #12（S3 误设 is_confirmed 致 S5 永远 409）即 P1 bug 打断 S6。
2. **变更优先于它影响的任务**：任务实现依赖未落地变更时，变更先做，否则按旧设计做后返工。"变更"指 doc/10 §四流程产出的任务集（非 `type:change` 标签）。
3. **同类按优先级 + 依赖**：P0>P1>P2；有依赖按拓扑序。

### 冲突仲裁

bug vs 变更争资源：bug 不修代价"功能持续坏"，变更不做是"设计待更新"。bug 代价更即时，故 bug 先。除非变更本身是修该 bug 根因，则变更先。

### 承载

无需新机制，标签组合表达：排序 = `优先级 asc, 类型(bug>change>feature), 依赖(无依赖优先)`。主 agent 派发前 `gh issue list --label "ready-for-agent"` 按此排序。

## 五、场景流程

### 场景 A：主 agent 派发任务 + 更新状态

1. `gh issue list --label ready-for-agent` 按优先级排序选第一个。
2. 派发子 agent：issue body（任务信息）+ 子 agent 读项目 CLAUDE.md（项目环境）= 契约单；主 agent 据任务性质设 model（机械->haiku，业务->sonnet）。
3. issue 标 `ready-for-agent` -> 开始时改 `in-progress`（或用 assignee 标记）。
4. 完成后 PR `Fixes/Refactors #N` -> 合并 -> issue 自动关闭。
5. progress.md「下一步」删该条（不追加完成记录，完成证据落 commit+issue+git）。

### 场景 B：新 session 接手 + 获取进展 + 派发 ⭐核心

三板斧（token 最省顺序）：
1. **读 progress.md**（<20 行）：当前焦点 + 下一步指针 + 阻塞。一眼知全局。
2. `gh issue list --state open`：核对开放工作（交叉验证，矛盾以 git/issue 为准，顺手改 progress.md）。
3. `git log --oneline origin/main | head`：最近合并。

从 progress.md「下一步」第一条开始，该 issue 详情即契约单输入。

### 场景 C：发现 bug 提交

开 issue（bug 模板，标 `type:bug,P{N},needs-triage`）-> triage reproduce 验证 -> 标 `ready-for-agent` -> 分配 -> 修 -> PR `Fixes #N` -> 关闭。bug 不进 progress.md（除非 P0/P1 阻塞当前焦点）。

### 场景 D：变更检测后加任务

走 doc/10 §四：发起 -> 追问澄清 -> 影响检测（清单）-> 记 D{N}+Updates -> 执行 -> 一致性验证。**变更流程不在本套 issue 体系内**，它的产物是 feature/bug/spike 类 issue（开 `type:feature/spike`），progress.md 加指针。change 本身不作为一个 issue 类型。

## 六、依赖与获取下一任务

### 依赖维护

GitHub 原生：issue 描述 `Depends on #N` / `Blocked by #N`，自动渲染。to-issues 用 sub-issue + blocking edge 原生链接（body 里 Parent/Blocked by 是 fallback）。

### 获取下一任务

```
gh issue list --label "ready-for-agent" --json number,title,labels --jq 'sort_by(.number)'
```

`ready-for-agent` = 依赖满足、立即可做。progress.md「下一步」1/2/3 指向这些（有序）。

### 了解进展 / 从哪开始

progress.md 是"从哪开始"入口，issues 是"做什么"详情，git 是"做了什么"历史。三者各司其职。

## 七、泛化（粒度无关）

后续项目不按 Story 拆时：

- **拆分粒度自由**：用 milestone 表达（Story/MVP/v1/模块 都行），不依赖固定标签。`gh issue list --milestone <X>` 过滤。
- **依赖用原生链接**：`Blocked by #N`，不依赖标签。spike 可 `unblocks #N`。
- **progress.md 只存当前工作块指针**，不假设粒度。
- **硬约束**：无论 Story/milestone 拆，每 issue 必须自包含 + 可独立验证（triage 的 ready-for-agent 标准"no human context"）。粒度自由，自包含是硬约束。

## 八、skill 规划

### 研究结论（to-issues + triage，借鉴思想不照搬）

**[to-issues](file:///Users/isuntaiyang/projects/soft_engineering/.claude/skills/to-issues/SKILL.md)**（任务拆解）：vertical slice（tracer bullet，贯穿所有层端到端）+ quiz user（粒度/依赖确认，迭代到批准）+ 模板（Parent/What to build/Acceptance/Blocked by）+ 依赖原生链接。约束：issue body 避免文件路径/代码（易过期）。

**[triage](file:///Users/isuntaiyang/projects/soft_engineering/.claude/skills/triage/SKILL.md)**（任务推进）：5 状态机（needs-triage/needs-info/ready-for-agent/ready-for-human/wontfix）+ agent brief（AGENT-BRIEF.md，=契约单范本）+ verify claim（bug reproduce）+ grill（打磨）+ out-of-scope（拒绝库）。

### 借鉴而不照搬的原因

- to-issues 命名像"提交问题"非"拆解任务"；输入是 conversation/单 issue，我们是 doc/ 多文件；模板缺关联 D{N}。借 vertical slice + quiz user，改名、改输入、改模板。
- triage 面向开源多外部请求（needs-triage/out-of-scope 重），单人+agent 场景过剩。借状态机简化版 + agent brief。

### 定稿：自建 2 个 skill

| skill | 职责 | 借鉴 |
|---|---|---|
| **`spec-to-tasks`** | 读 doc/（01 用户故事 + 06 流程清单为主，02/04 补实现，03 定约束）-> vertical slice -> quiz user（粒度/依赖确认）-> 发布 issue（§三模板，任务自身信息） | to-issues（vertical slice + quiz user + 依赖链接） |
| **`task-advance`** | 接手三板斧 -> 抽契约单派发 -> 验证（code review + 本地测试 + CI 监控 + 合并 + 冲突解决）-> 更新 progress/状态 | triage（状态机简化版 + agent brief） |

> issue 模板（bug/feature/spike/chore/docs）用 `.github/ISSUE_TEMPLATE/*.yml` form，项目初始化创建，非 skill（主 agent 据类型选 form，CLAUDE.md 一句规范）。

### skill 1：`spec-to-tasks`（任务拆解，跨项目复用）

**输入**：doc/ 下的设计文档（01 用户故事 + 06 流程清单为 vertical slice 主源，02 数据模型/04 API 补实现，03 架构定约束）。doc/ 与 PRD 等价（见 multi-agent-chat issue#1，PRD 含 User Stories + Implementation Decisions，即设计文档集）。

**流程**（借 to-issues §1-5）：
1. 读 doc/ 相关章节（主动读，非依赖 conversation 上下文）。
2.（可选）探索 codebase，用 domain glossary，尊重 ADR。
3. Draft vertical slices：每 slice 贯穿所有层（schema/API/UI/tests）端到端、可独立验证。06 的「步骤|用户动作|系统行为」表天然是 vertical slice 雏形。
4. **Quiz user**（to-issues §4）：展示编号清单（Title/Blocked by/覆盖的用户故事），问粒度/依赖/合并拆分，迭代到批准。
5. 发布：按依赖序（blocker 先）发布 issue（§三模板），用 GitHub 原生 sub-issue + blocking edge 链接依赖。标 `ready-for-agent`。

**与 to-issues 差异**：改名（spec-to-tasks 非 to-issues）；输入是 doc/ 多文件非单 plan；模板加「关联决策 D{N}」。

### skill 2：`task-advance`（调度推进 + 验证 + 合并，跨项目复用）

**核心**：主 agent 的调度工作流，含接手、派发、**验证闭环**、CI 监控、合并、冲突解决。这些是跨项目重复的固定流程，skill 化后不在每个项目 CLAUDE.md 重写。

**分工（CLAUDE.md 触发指针 + skill 详情）**：task-advance 是主 agent 调度工作流的详情说明书。因跨项目复用 + 不占每会话上下文而 skill 化；CLAUDE.md 保留一句触发指针（"主 agent 接手/派发/合并时调 task-advance skill"）保证不漏调。即 CLAUDE.md 是瘦指针层（自动加载、不漏），task-advance 是详情层（按需加载、复用）。与文档治理（index 指针 + 子文件详情）同构。

**流程**：

#### 阶段 1：接手（新 session）

三板斧（token 最省顺序，见 §五场景 B）：
1. 读 progress.md（<20 行，当前焦点+下一步指针+阻塞）。
2. `gh issue list --state open` 核对开放工作（矛盾以 git/issue 为准，顺手改 progress.md）。
3. `git log --oneline origin/main | head` 看最近合并。
从 progress.md「下一步」第一条开始。

#### 阶段 2：派发

1. `gh issue list --label ready-for-agent` 按优先级选第一个。
2. 派发子 agent：issue body（任务信息）+ 子 agent 读项目 CLAUDE.md（项目环境）= 契约单。
3. 主 agent 据任务性质设 model（机械->haiku，业务->sonnet，见下方模型选择原则）。
4. issue 标 `in-progress`（或 assignee）。

> **模型选择原则**（task-advance 承载，跨项目通用）：机械层（ORM model/CRUD repo/薄路由/migration，给定 spec 输出确定、少判断）-> haiku；业务层（AppSvc 级联/状态机/事务边界/边界 case，需判断）-> sonnet；reviewer 固定 sonnet（`.claude/agents/code-reviewer.md` frontmatter，铁律违反要靠它抓，不降 haiku）。主 agent 读契约单时判断本任务机械还是业务，据此设 model。模型选型是派发时调度决策，不进 issue、不进 CLAUDE.md（非任务固有属性）。

#### 阶段 3：验证闭环（子 agent 完成后）

子 agent 完成 -> 主 agent 跑验证闭环（不直接信子 agent 自报）：

- **本地测试**：`make test`（或项目等价）全绿。子 agent 在 worktree 跑过，主 agent 合并前可复跑关键项。
- **lint**：`make lint`（ruff check + format check）。
- **code review**：派 code-reviewer agent（独立于实现 agent）审 PR diff + 铁律速查单，产报告（`gh pr review --comment`）。
  - P0/P1 -> 转回实现 agent 修 -> push 重跑 CI -> review 复审。
  - 无 P0/P1 -> 进阶段 4。
  > code-reviewer 已 sonnet（`.claude/agents/code-reviewer.md`），只读不改。
- **迁移检查**（若 PR 含 alembic 迁移）：CI 跑 alembic 空库演练；人工看 upgrade/downgrade。

#### 阶段 4：CI 监控 + 合并

CI 监控（项目级规范，见 CLAUDE.md CI 段）：
```
gh pr checks <N> --watch && echo "ALL PASS" || echo "HAS FAIL"
```
`--watch` 阻塞到终态退出即通知；exit=0 全 pass，非 0 有 fail。无需手写轮询。

合并（CLAUDE.md §十二已有规范，引用不重写）：
- **squash 合并**：`gh pr merge --squash --delete-branch`（一个任务 = main 上一条提交，线性历史）。
- **合并是主 agent 职责**，子 agent 不自行合并。
- **合并顺序 = 依赖顺序**：同并行窗口一个合完、main 更新后另一个 rebase 再合，不并行合。
- PR 含 `Fixes #N` -> issue 自动关闭。
- 合并后删 worktree（`git worktree remove`）+ 删分支。

#### 阶段 5：冲突解决规范

- **子 agent 开工前**：`git fetch && git rebase origin/main`（基要最新，CLAUDE.md §十二）。
- **合并前 rebase 冲突**：合第二个任务前先 `git rebase origin/main`，冲突按 resolving-merge-conflicts 思想（看状态->查 primary source->逐 hunk 保留双方意图->跑 checks->finish），**不用 `--abort`**。
- **alembic 多 head 冲突**（并行迁移）：方案 A `uv run alembic merge -m "merge X Y" <revX> <revY>` 生成合流迁移；方案 B 手改 down_revision。`uv run alembic upgrade head` 验证无误后才合并。**迁移合并由主 agent 执行**，不让子 agent 自合。

#### 阶段 6：更新状态

- issue 自动关闭（`Fixes #N`）。
- progress.md「下一步」删该条（不追加完成记录，证据落 commit+issue+git）。
- 若有新发现的 bug/任务，开 issue（走 §五场景 C），progress.md 加指针。

> **task-advance 与 triage 的关系**：triage 是"单 issue 状态机推进"（issue 视角，含外部请求评估），task-advance 是"主 agent 工作流"（会话/队列视角，含验证合并）。单人+agent 场景 task-advance 更贴合，借 triage 的状态机简化版 + agent brief。

### 模板非 skill 化

issue 模板用 `.github/ISSUE_TEMPLATE/*.yml` form，项目初始化创建（非每次判断、非 skill）。主 agent 开 issue 时 `gh issue create` 选 form。CLAUDE.md 补一句选型规范（据类型选 form）。

## 附：方案总结（项目无关）

1. **三层分工**：progress.md（指针，<20 行）/ issues（详情）/ git（历史），单一写入。
2. **标签 3 维**：triage 5 状态 + 5 类型 + 4 优先级（无 type:change）。兼容 triage skill。
3. **issue 模板**：GitHub form，自包含，借鉴 to-issues 模板；**不放项目绑定字段**（铁律/复用件/模型归 CLAUDE.md）。
4. **契约单 = issue + CLAUDE.md**：子 agent 读 issue（任务信息）+ 项目 CLAUDE.md（项目环境），不重复。
5. **并存排序**：bug(P0/P1) > 变更(影响下游) > 任务，优先级+类型+依赖标签组合。
6. **2 个自建 skill**：`spec-to-tasks`（拆解，借 to-issues）+ `task-advance`（调度+验证+合并，借 triage）。模板用 GitHub form。
7. **粒度无关**：用 milestone+原生依赖，不绑 Story。
