# 飞书卡片验证（schema 2.0 集成行为）

> 飞书卡片 schema 2.0 集成验证事实。所有结论均用真实飞书 API 实测（推送 + 回调 + update_card），非文档推测。
> 卡片样式见 `doc/09_卡片交互样式记录.md`；opencode 验证见 `doc/13_原型验证/opencode验证.md`。

## 试错记录（踩坑备忘，避免重蹈覆覆辙）

### V1. form_submit 按钮不能带 behaviors（关键坑）
- **现象**：form 内 button 设 `action_type: form_submit` + `behaviors: [{type:callback,...}]`，飞书报错 `230099 there is no submit button in the form container`。
- **根因**：form_submit 按钮的回传走 form_value 机制，不能再配 behaviors（behaviors 是 form 外回传按钮用的）。两者互斥。
- **正确做法**：form_submit 按钮只留 `tag/name/text/type/action_type`，不带 behaviors。区分多个 submit 靠 `name`（Service 读 `event.action.name`）。
- **迷惑点**：form 内只放 input + form_submit 按钮时，偶发不报错（飞书容错），但加 multi_select/date_picker 就必报错。所以**一律不带 behaviors**。

### V2. message_id 路径（FIX-1 修正）
- **错误**：FIX-1 写 `payload.get("open_message_id")` 取顶层 -> 恒空，update_card 坏。
- **正确**：`payload["event"]["context"]["open_message_id"]`。实测回调结构证实。
- **完整回调取法**：
  ```python
  event = payload["event"]
  message_id = event["context"]["open_message_id"]   # 刷新用
  action_value = event["action"].get("value", {})    # form 外按钮自定义回传
  form_value = event["action"].get("form_value", {}) # form 内组件输入值
  btn_name = event["action"].get("name", "")          # 区分 form_submit 按钮
  action_id = action_value.get("action_id")            # 仅 form 外按钮有
  ```

### V3.（已删除，序号保留）
- 原内容为 web-clipper 抓飞书文档站的调研踩坑，与卡片集成验证无关，已移除。序号保留以维持 V1-V9 引用稳定。

### V4. 飞书无原生"全选/反选"组件
- **查证**：飞书交互组件清单只有 button/input/checker/date_picker/select/multi_select 等，无"全选/反选"按钮。checker 是单个勾选（每 checker 独立状态），multi_select 是下拉多选。
- **S4B 后置全不选方案**：加 form 外「全不选」回传按钮（behaviors callback），点了 Service 收到回调 -> update_card 把所有 checker checked=false 刷新。不是飞书原生联动，是 Service 刷卡。
- **限制**：飞书 button 无法联动改其他组件状态，只能 update_card 整张卡刷新。

### V5. 旧版卡片（config+elements）不支持 form
- **查证**：用旧版格式推 form 容器，飞书 200 但 content 降级成"请升级至最新版本客户端"文本（form 不渲染）。
- **根因**：form 是 schema 2.0 特性，旧版不支持。
- **结论**：要用 form/input/date_picker/checker/multi_select，必须全局升 schema 2.0（`{"schema":"2.0","body":{"elements":[...]}}`）。

### V6. update_card 刷新 schema 2.0 可用
- **查证**：`PATCH /im/v1/messages/{message_id}` body `{"content": <schema2.0卡片JSON>}`，返回 200 code 0，卡片实时刷新。
- **注意**：飞书回调 token 有效期 30 分钟、最多更新 2 次（走回调 token）。但直接 PATCH message（用 app token）不受此限--本项目 update_card 用 app token 直接 PATCH，实测可多次刷新。

### V7. checker 回传是 bool、date_picker 是日期字符串、multi_select 是数组
- form_value 各组件类型：
  - checker：`{name: true/false}`
  - date_picker：`{name: "2026-07-24 +0800"}`（含时区）
  - multi_select：`{name: ["v1","v2"]}`
  - input：`{name: "用户输入文本"}`
- Service 解析时按类型取。

### V8. form_submit 按钮靠 name 路由（非 action_id）
- form_submit 按钮无 value/action_id，Service 必须按 `event.action.name` 路由（如 btn_pass/btn_reject/confirm_btn/btn_activate/btn_defer）。
- form 外按钮按 `action.value.action_id` 路由（如 story1_确认方案/story5_确认日终总结/story6_已阅/story8 全不选）。
- **两类按钮路由策略不同**，feishu_card.py 要分别处理。

### V9. 客户端版本要求
- schema 2.0 需飞书客户端 **7.20+**。低于此版本卡片内容显示兜底"请升级"提示（标题仍显示）。
- 本项目实测 form/checker/date_picker 全部正常工作，客户端满足要求。

## 关键验证事实（FACT）

### D27. 飞书 webhook 回调响应体可同步返卡
- **类型**：FACT（e2e 实测验证，非选型，无权衡）
- **事实**：飞书卡片交互回调（card.action.trigger）的 webhook 响应体可同步返回新卡片：`{"toast":{"type":"success","content":"..."},"card":{"type":"raw","data":<schema2.0 卡片 dict>}}`。点击后卡片立即刷新，无需二次调 update_card。
- **证据**：e2e S5 实测（PR #31），webhook 响应体含 card.data 立即刷新；confirm 幂等（PR #32）保证重复点击同步返终态卡。

### D28. 飞书 GET 查询 schema 2.0 卡片降级
- **类型**：FACT（e2e 实测，外部 API 客观行为）
- **事实**：`GET /im/v1/messages` 查询 schema 2.0 卡片时返回 legacy 占位（非实际卡片 JSON），故卡片刷新态无法通过 GET 回查，只能人工视觉确认。
- **证据**：e2e 期实测，memory 记录「飞书 schema2 GET 降级」。
