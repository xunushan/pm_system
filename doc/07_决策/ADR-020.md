# ADR-020: opencode 进程模型--全局单进程 + 多 session

OpenCode 改为全局单进程 + 多 session 模型，替代原先每 session 独立进程的多进程模型。

## Context

OpenCode 原设计为每 session 独立进程（基于早期多进程设想，见 ADR-015 的进程生命周期描述）。这提供强隔离但消耗大量内存，且使跨 session 通信不可能。PR #19 重写 opencode.py 采用方案 B（全局单进程 + 多 session，类级单例 _proc），不同 workspace 用不同 session_id，全局进程保留服务所有 workspace。现状代码缺"退 session"方法：3 次不通过时应调 opencode 的 DELETE /session/:sessionID 退该 task session，但 opencode.py 无此方法。

## Decision

以方案 B 为准：OpenCode 作为单一全局进程运行，在该进程内创建多个 session，session 共享进程但通过 session 上下文隔离。
- 3 次不通过调 DELETE /session/:sessionID 退该 task session，全局 serve 进程不退出、保留服务其他 workspace。
- 用户用 session_id 在本地接管，"/pm 确认完成"后系统重新建/复用 session 驱动后续任务。
- 代码缺口：opencode.py 需补 delete_session(session_id) 方法，3 次不通过时调用。
- 同时修正 ADR-015 中过时的"进程退出/不同端口重启"描述（核心决策不变）。

## Alternatives Considered

### 每 session 独立进程（原方案，早期多进程设想）
- Rejected: 高内存占用、无共享状态、session 启动慢；当前工作负载不需要此隔离。

## Consequences

- 内存使用更低，session 启动更快。
- session 可共享进程内缓存和状态。
- 进程崩溃影响所有 session，必须优雅处理。
- 影响：doc/06 Story 4A 步骤 8/9 改写；doc/09 记 S4A 场景 4；opencode.py 补 delete_session。
