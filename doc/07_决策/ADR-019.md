# ADR-019: 飞书卡片全生命周期归 Service 层

Service 层拥有飞书卡片的完整生命周期（构建 + 发送 + 回调路由 + 点击后刷新），消歧"卡片归 Skill 还是 Service"的矛盾。

## Context

设计文档对"卡片归谁"自相矛盾：doc/03 §1.2 职责表写"生成飞书卡片 ✅ Skill / ❌ Service"，而 doc/05 §1.3 下沉表/§8.1 归属表、doc/06 操作流程、doc/04 回调模块写"卡片 patch/验收卡/衔接卡 -> Service"。子 agent 读到 §1.2 的"❌ Service"误以为卡片是 Skill 的活，导致多个 Story 无 Service 推卡入口、缺构建器。叠加"开发期 Skills 不实现"，"卡片归 Skill"等于"开发期没人推卡"。根因是两张表把同一能力分给不同主体，从未消歧。

## Decision

Service 拥有卡片全生命周期：构建（build_*_card）+ 发送（send_card/update_card/send_file）+ 回调路由（/webhook/feishu/card action_id 硬编码）+ 点击后刷新（update_card 重建整张卡）。Skill 只负责调用 Service 的推卡方法（讨论态卡片的初始推送可由 Skill 触发，但构建逻辑在 Service）。这是 ADR-002 判定标准"纯确定性逻辑（含 HTTP 调用）-> Service"的显式化，卡片构建/发送/刷新均为模板填充 + HTTP，无 LLM。

## Alternatives Considered

### 卡片归 Skill（doc/03 §1.2 旧表述）
- Rejected: 把同一能力分给 Skill 和 Service 两处自相矛盾；叠加"开发期 Skills 不实现"导致开发期无 Service 推卡入口，e2e 无法触达。

## Consequences

- 卡片行为和格式的单一真相源在 Service。
- 所有卡片回调处理后必须 update_card 重建整张卡（按钮灰化/反转/消失），禁止"点击后卡片不变"。message_id 优先从飞书回调 payload 顶层取，fallback 从 action_value 取。
- 按钮置于对应任务/项旁（per-item action），禁止所有按钮挤在单个 action 块；卡片状态在卡片上可见体现。
- 开发期补偿：Skills 仅占位期间，凡文档说"Skill 推卡"的场景必须有等价 Service 推卡方法兜底。
- 影响：doc/03 §1.2 同步修正；CLAUDE.md 铁律加第 11 条；代码补 5 个构建器 + 4 个 Service 推卡入口 + 全回调 update_card + 按钮挨任务样式。
