# ADR-018: drafts 表用于确认前存储

恢复 drafts 表，用于 pm-plan 生成规划数据的确认前存储，规避飞书回调数据量限制。

## Context

一个目标的完整规划（目标 + 4 专题 + 24 阶段 + 72 任务）数据量可达几十 KB（约 30-60KB）。飞书卡片回调（card.action.trigger）请求体上限约 30KB，确认按钮回调若携带完整规划数据可能超限失败。此前第二轮决策曾废弃 drafts 表（理由是"草案进看板"is_draft 机制复杂），但废弃后 pm-plan 生成的规划数据在确认前无处存储（Service 收到回调时拿不到 Skill 内存数据）。

## Decision

恢复 drafts 表，用于 pm-plan 生成规划数据的确认前存储（纯存储，不同步展示）。
- 数据流：pm-plan 生成规划->调 Service 写 drafts（content 存完整规划 JSON）->总览卡片按钮 action_value 只含 draft_id->确认回调只传 draft_id->Service 用 draft_id 读 drafts->写正式表->删 drafts。
- drafts 纯存储，不进 H5 页面/多维表格。
- is_draft 仍废弃：规划数据存 drafts 不存正式表，不污染正式表，不恢复 is_draft 机制。
- SQLite TEXT 字段无大小限制，drafts 存几十 KB 规划数据无问题。

## Alternatives Considered

### 废弃 drafts 表（第二轮决策，已修正）
- Rejected: 废弃后 pm-plan 生成的规划数据在确认前无处存储，Service 收到回调时拿不到 Skill 内存数据；且飞书回调 30KB 限制下完整规划数据无法直接传递。

## Consequences

- 无论卡片复杂度如何，回调始终低于 30KB（只传 draft_id）。
- 需维护 drafts 表并定期清理。
- 确认后增加一次额外查询步骤。
- 通用原则：所有飞书卡片设计避免一次性传大数据，回调只传标识符（draft_id、task_id 等）。
- 影响：01/06 Story 1 改写；02 恢复 drafts 表；04 drafts 接口恢复（POST/GET/PUT/DELETE /api/v1/drafts），plans/confirm 用 draft_id 读 drafts 写正式表。
