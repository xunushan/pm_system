## 一、总览

Service 代码开发完成（9/9 Story 合并，495 单测全绿）后，端到端验证发现**全 Story 卡片交互不合格**。复盘确认根因不是"agent 偷懒"或"缺决策"，而是：

1. 设计文档自相矛盾，子 agent 读到打架指令而选错边（L1）
2. 测试金字塔缺顶层，495 绿不等于系统能用（L2）
3. 卡片结构测只查 dict 键，飞书拒收才发现（L3）
4. 基础设施最后才配，全程无真实反馈信号（L4）
5. 测试脚本绕真实入口 + 手改 DB，掩盖失败（L5）
6. 异步副作用被 mock 掉且不断言（L6）

下阶段每个动作必须对照本文件检查是否重蹈覆辙。

## L1. 设计文档自相矛盾致隐性偏离

**现象**：S2/S3/S6 没有建 Service 推卡入口，S1/S2/S3/S4B/S6 缺卡片构建器。子 agent 认为"卡片是 Skill(CardKit) 的活"，Service 不该碰。

**根因（证据）**：设计文档对"卡片归谁"给了打架指令——
- doc/03 §1.2 职责表：「生成飞书卡片 ✅ Skill / ❌ Service」「CardKit->飞书 API 发送/更新」
- doc/05 §1.3 下沉表：「卡片 patch -> ScheduleAppSvc(Service)」「验收卡片 -> AgentAppSvc(Service)」
- doc/05 §8.1 归属表：「智能体下发/回调/验收卡片 -> Service，LLM=❌」
- doc/06 操作流程：「Service 调飞书 API 发验收卡/patch 卡/衔接卡」
- CLAUDE.md §11（框架层，§8 明令"遵从"）：`app/clients/feishu.py` 列为 Service 共享件

两张表把同一能力分给不同主体，从未消歧。叠加"开发期 Skills 不实现"（CLAUDE.md §8.4），"卡片归 Skill"等于"开发期没人推卡"。

**机制对策**：
- doc/07 D25 显式裁决：Service 拥有卡片全生命周期（build+push+callback+update_card）；Skill 只调 Service 推卡方法。
- 同步修正 doc/03 §1.2 表行，消除「❌ Service」一刀切。
- CLAUDE.md 铁律加第 11 条。
- **开发期 Skill 缺位补偿**：凡文档说"Skill 推卡"的场景，必须有等价 Service 推卡方法兜底，否则 e2e 无法触达。

## L2. 495 测试全绿 ≠ 系统能用（测试金字塔缺顶层）

**现象**：CI 持续绿，但端到端连一张飞书卡都发不出去。把"提交门绿"错当成"完成信号"。

**根因（证据）**：CLAUDE.md §6 自定义测试边界为「单元（级联/状态机/executor 推断）+ 集成（API + DB）」「不碰真实 DB」。实测 495 个测试：飞书 HTTP 全 mock httpx（feishu.py:7 注释「测试 mock httpx，不真连飞书」）、opencode 全 mock、redis 用 fakeredis。**测试体系有意只管确定性逻辑 + API/DB 事务，不管外部交互真实性**。这是设计意图，不是 bug——但意味着"系统能否发卡"从未被任何测试回答过。

**机制对策**：补测试金字塔顶层，分层见下「测试三分」。

## L3. 卡片结构测只查 dict 键，飞书拒收才发现

**现象**：v1 端到端时飞书返回 230099（卡片格式错），但 495 单测全绿。`test_feishu_client.py:147` 注释自承"卡片结构问题曾经是端到端发现的"。

**根因（证据）**：`build_*_card` 的结构校验只检查返回 dict 的键名/标签，**不校验飞书是否接受这个结构**。这是单测能做（不花真实推送的钱）却没做的真缺口。

**机制对策**：补一道"契约测试"层——用飞书官方卡片 JSON Schema 校验 `build_*_card` 输出。不依赖真实飞书，可进 CI 硬门禁。

---

## L4. 基础设施（webhook/redis/app_id）最后才配

**现象**：开发期全程没配飞书 app_id、redis、webhook 隧道，直到端到端才搭。全程无"连不上外部"的早期信号。

**根因（证据）**：CLAUDE.md §6 门禁 = ruff + pytest + alembic，三者全 mock 外部依赖，CI 从不碰真实飞书/redis/opencode。§8.4 开发期 Skills 仅占位、H5 仅骨架，没有真实入口会被真实用户触发。§5 Story 流程 model->migration->repo->service->api->test 没有一步"接通外部"。基础设施天然被推到 e2e 才需要。

**机制对策**：项目级一次性配置（Story0 就配好 webhook/redis/app_id）+ 加一道"基础设施就绪"冒烟门（`tests/e2e/test_infra.py`，断言 token 能取、卡能推、回调能到、opencode 能起、redis 能 ping）。

---

## L5. 测试脚本绕真实入口 + 手改 DB，掩盖失败

**现象**：v1 测试脚本直接调 `plans/confirm` 跳过 `POST /drafts`（导致 draft 表空，误判为"代码缺 draft 流程"）；手动改 task status / executor=human 掩盖真实链路断点。

**根因（证据）**：测试计划未约束"数据准备必须走真实入口""禁止手改 DB"。脚本为"快"走捷径，反而把真 bug（S4A 主任务不下发、S5 message_id 空）埋了。

**机制对策**：e2e 用例规范强制 7 字段（前置/数据准备方式/步骤/卡片预期/DB预期/**禁止项**/通过判据）。数据准备必须经真实 API 入口（drafts->plans->schedules->daily 全链路），禁止直接写 DB。禁止项显式写"不得手改 status/executor"。

## L6. 异步副作用被 mock 掉且不断言

**现象**：S5 点击"标记完成"后卡片没刷新，但集成测试显示"通过"。

**根因（证据）**：`test_story5_daily_summary.py:375,409` 用 `patch.object(..., "update_card")` 把刷新 patch 成 no-op 防报错，**但从不 `assert_called()`**。webhook 路由用 `background_tasks.add_task(refresh_summary_card_async)` 异步刷新，TestClient 同步执行了它，但测试只验主请求的 DB 变化，不验异步副作用是否发生。更深的 bug：`build_daily_summary_card` 的按钮 value 不含 `message_id`（feishu.py:228-265），而回调 `feishu_card.py:132` 从 `action_value.get("message_id", "")` 取——恒为空串，`update_card("", card)` 指向无效 URL，刷新实际坏掉。

**机制对策**：e2e 必须断言异步副作用真实发生（卡片真刷新 + HTTP 返回 code=0）。集成测试若 patch 了 update_card，必须 `assert_called_once()`。

## 二、测试三分（概念划分，避免混淆）

> 本节是 L2 的落地规范。术语在不同团队有重叠用法，本项目给它们明确分工，下阶段统一使用。

### 三层职责

| 层 | 目录 | 回答的问题 | 依赖 | 进 CI 硬门禁？ | 跑的时机 |
|----|------|-----------|------|--------------|---------|
| **集成测试** | `tests/integration/` + `tests/unit/` | Service 内部逻辑串得通吗（路由+事务+DB+级联）| 全 mock（内存 SQLite、fakeredis、mock httpx）| ✅ 是 | 每 PR / 每 commit |
| **冒烟测试** | `tests/e2e/test_infra.py` | 系统活着吗、外部链路通吗 | 真实飞书+opencode+redis | ❌ 否 | 部署/起服务后 |
| **端到端（e2e）** | `tests/e2e/`（除 infra）| 每个功能对不对（逐 Story 真测）| 真实飞书+opencode+redis | ❌ 否 | 发布前（手动）|

### 关键区分

**集成测试 vs e2e 的 webhook**（本项目易混点）：
- 集成测试里的 webhook：`TestClient` 自己手拼 dict 发 POST，飞书**不在场**，验"Service 处理这个 payload 对吗"。
- e2e 里的 webhook：**真实飞书**点按钮后发的，验"飞书->ngrok->Service 整条外部链路通且业务对"。
- 集成测试 webhook 绿 ≠ 真能收飞书回调。这是 v1 盲区。

**冒烟 vs e2e**：
- 冒烟验"通不通"（链路连通性），不验业务正确性。快。
- e2e 验"对不对"（链路通 + 业务正确）。慢。
- 冒烟是 e2e 的**前置闸**：冒烟挂了 e2e 必白跑。先冒烟，绿了再跑 e2e。
- 类比：冒烟=通电看冒不冒烟；e2e=逐按钮测每个功能。

### 行业标准（e2e 是发布门，不是提交门）

来源：
- Martin Fowler《TestPyramid》：底层多单测，顶层少量 e2e（martinfowler.com/bliki/TestPyramid.html）
- Google Testing《Just Say No to More E2E Tests》：e2e 慢/flaky/维护贵，不要堆叠当主力（testing.googleblog.com/2015/04/just-say-no-to-more-end-to-end-tests.html）
- Google TotT《What Makes a Good E2E Test》：e2e 是均衡测试关键部分但要审慎
- Kent C. Dodds《Write tests. Not too many. Mostly integration.》：主力是集成（kentcdodds.com/blog/write-tests）

行业实践：e2e 不放每个 PR 的硬门禁（慢+flaky 会拖垮迭代），放**部署到 staging / 夜间定时 / 发布前**。但宣布"开发完成"时 e2e 必须绿——这是**发布门**。

本项目落地（依赖真实飞书/opencode/redis，全是不稳定外部依赖）：
1. e2e 不进 CI 硬门禁（§12 现有 CI=ruff+pytest+alembic 不变）。
2. 补"契约层"进 CI：飞书官方卡片 Schema 校验 `build_*_card` 输出（不花真实推送的钱挡结构错误）。
3. 完成信号明确化：每个含卡 Story 完成 = unit/integration 绿（CI）+ 契约绿（CI）+ e2e 该 Story 用例真跑过（手动）。三者齐全才算完成。

## 三、教训应用检查清单

下阶段每个 PR / 每次测试，对照检查：

- [ ] 本 Story 的卡片归属是否按 D25（Service 全生命周期）实现？未依赖不存在的 Skill 推卡
- [ ] 是否在 CI 之外补了 e2e 真测？未把提交门绿当完成信号
- [ ] `build_*_card` 是否过了飞书官方 Schema 契约校验？
- [ ] 基础设施（飞书/redis/opencode/webhook）是否就绪并过了冒烟？
- [ ] e2e 数据准备是否走真实 API 入口？未手改 DB
- [ ] 异步副作用（update_card 刷新、opencode 下发）是否真实断言？未只 mock 不断言
