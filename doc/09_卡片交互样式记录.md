# 卡片交互样式记录（schema 2.0 实证）

> 每个 Story 跑通后追加。写代码时照此实现，避免交互偏离。
> 所有 JSON 已用真实飞书 API 验证可推送/可回调/可 update_card 刷新。

## 通用规则（全 Story 适用）

- **卡片格式**：schema 2.0（`{"schema":"2.0","header":{...},"body":{"elements":[...]}}`），旧版 config+elements 废弃
- **两类按钮**（关键，别混淆）：
  - form 外回传按钮：`{"tag":"button","text":{...},"type":"...","behaviors":[{"type":"callback","value":{"action_id":"...","<id>":"..."}}]}`，action_id 从 `event.action.value.action_id` 取
  - form 内提交按钮：`{"tag":"button","name":"<btn_name>","text":{...},"type":"...","action_type":"form_submit"}`，**不能带 behaviors**，靠 `event.action.name` 区分动作
- **回调取法**：
  - message_id：`event.context.open_message_id`
  - action.value：`event.action.value`（form 外按钮的自定义回传）
  - form_value：`event.action.form_value`（form 内组件输入，key 是组件 name）
  - btn name：`event.action.name`（区分 form 内哪个 submit 被点）
- **点击后必须 update_card**：飞书不自动置灰/移除按钮，Service 收到回调后必须 PATCH message 刷新卡片，否则可重复点击（v1 问题）
- **update_card**：`PATCH /im/v1/messages/{message_id}`，body `{"content": <新版卡片JSON>}`，schema 2.0 可正常刷新（实证）
- **链接跳转**：用 behaviors `{"type":"open_url","default_url":"..."}`（按钮跳转）或 markdown `[文字](url)`（文字链接）

```json
  ]}
```

- 标题蓝色（blue，待操作）

- 对勾选的专题，锁定每个的第 1 个未开始阶段（强约束）

### 实证
- checker 勾选回传 ✅（form_value.theme_<id> = bool）
- date_picker 回传 ✅（form_value.dl_<theme_id> = 日期含时区）
- form_submit 按钮不能带 behaviors ✅
- patch A->B update_card ✅
- 用户确认方案：checker 勾选专题 + date_picker 选 deadline，managed 默认全托管走配置页

### 实现注意（写代码时）
- Service 路由要按 `event.action.name` 识别 form_submit 按钮（next_btn/confirm_btn），不能只靠 action_id（form_submit 无 value/action_id）
- form_value 里 checker 是 bool、date_picker 是日期字符串、multi_select 是数组
- 名额校验（≤3）在 confirm 阶段做

## S3 当日计划推送

### 卡片类型：form + checker（候选任务 + 前置，两组独立勾选）

```json
      {"tag": "checker", "name": "task_<id2>", "text": {"tag": "plain_text", "content": "代码实现与工程化：部署 RSSHub（知识获取/阶段1）[人]"}},
```

- 事务后异步：opencode 执行勾选前置 + 若有 agent 任务启动 opencode serve

## S4A 智能体执行验收（4 个场景）

### 卡片类型：form + 两个 submit（验收通过/需要修改）+ input feedback；场景3纯提醒无按钮；场景4确认完成推卡

```json
  "body": {"elements": [
    {"tag": "markdown", "content": "**任务：<task_name>**\n\n**产出文件：**\n· <file1>\n· <file2>"},

      {"tag": "button", "name": "btn_pass", "text": {"tag": "plain_text", "content": "验收通过"}, "type": "primary", "action_type": "form_submit"},
      {"tag": "hr"},
      {"tag": "markdown", "content": "**若需修改，请填写反馈后点「需要修改」：**"},

    ]}
  ]}
}
```
- form 内两个 submit 按钮，靠 name 区分（btn_pass/btn_reject），不能带 behaviors
- feedback input 收修改建议

### 场景1：点「验收通过」后（update_card 刷新）

- **不是"正在重试（1/3）"**（修正：反馈是下发智能体调整，非实时重试提醒）
- 显示用户填的 feedback

# 验证/试错记录（踩坑备忘，避免重蹈覆辙）

> 以下结论均用真实飞书 API 实测（推送 + 回调 + update_card），非文档推测。

## V1. form_submit 按钮不能带 behaviors（关键坑）
- **现象**：form 内 button 设 `action_type: form_submit` + `behaviors: [{type:callback,...}]`，飞书报错 `230099 there is no submit button in the form container`。
- **根因**：form_submit 按钮的回传走 form_value 机制，不能再配 behaviors（behaviors 是 form 外回传按钮用的）。两者互斥。
- **正确做法**：form_submit 按钮只留 `tag/name/text/type/action_type`，不带 behaviors。区分多个 submit 靠 `name`（Service 读 `event.action.name`）。
- **迷惑点**：form 内只放 input + form_submit 按钮时，偶发不报错（飞书容错），但加 multi_select/date_picker 就必报错。所以**一律不带 behaviors**。

## V2. message_id 路径（FIX-1 修正）
- **错误**：FIX-1 写 `payload.get("open_message_id")` 取顶层 -> 恒空，update_card 坏。
- **正确**：`payload["event"]["context"]["open_message_id"]`。实测回调结构证实。
- **完整回调取法**：
  ```python
  event = payload["event"]
  message_id = event["context"]["open_message_id"]   # 刷新用
  action_value = event["action"].get("value", {})    # form 外按钮自定义回传
  form_value = event["action"].get("form_value", {}) # form 内组件输入值
  btn_name = event["action"].get("name", "")          # 区分 form_submit 按钮
  action_id = action_value.get("action_id")            # 仅 form 外按钮有
  ```

## V3. 飞书文档站抓不到的根因（web-clipper 无罪）
- **现象**：web-clipper 抓飞书文档只拿到 ~1800 字节空壳。
- **查证过程**：playwright 加载飞书文档站，JS/CSS 全 200 加载成功，但 body 文本仅导航栏（279 字）。监听 XHR 发现文档内容 API `document_portal/v1/document/get_detail` 返回 `code:41404 文档不存在`。
- **根因**：URL 的 document_id 是编的（不存在），不是 SPA 渲染问题、不是反爬。web-clipper 本身可用（抓 GitHub 等静态站正常）。
- **教训**：飞书文档 URL 的 document_id 段不能编。用 anysearch 搜正确 URL（搜到的 result URL 可直接用）。
- **飞书文档站是 SPA**：内容靠 `get_detail` API 异步加载，正确 URL 下 playwright 能等到渲染。

## V4. 飞书无原生"全选/反选"组件
- **查证**：飞书交互组件清单只有 button/input/checker/date_picker/select/multi_select 等，无"全选/反选"按钮。checker 是单个勾选（每 checker 独立状态），multi_select 是下拉多选。
- **S4B 后置全不选方案**：加 form 外「全不选」回传按钮（behaviors callback），点了 Service 收到回调 -> update_card 把所有 checker checked=false 刷新。不是飞书原生联动，是 Service 刷卡。
- **限制**：飞书 button 无法联动改其他组件状态，只能 update_card 整张卡刷新。

## V5. 旧版卡片（config+elements）不支持 form
- **查证**：用旧版格式推 form 容器，飞书 200 但 content 降级成"请升级至最新版本客户端"文本（form 不渲染）。
- **根因**：form 是 schema 2.0 特性，旧版不支持。
- **结论**：要用 form/input/date_picker/checker/multi_select，必须全局升 schema 2.0（`{"schema":"2.0","body":{"elements":[...]}}`）。

## V6. update_card 刷新 schema 2.0 可用
- **查证**：`PATCH /im/v1/messages/{message_id}` body `{"content": <schema2.0卡片JSON>}`，返回 200 code 0，卡片实时刷新。
- **注意**：飞书回调 token 有效期 30 分钟、最多更新 2 次（走回调 token）。但直接 PATCH message（用 app token）不受此限--本项目 update_card 用 app token 直接 PATCH，实测可多次刷新。

## V7. checker 回传是 bool、date_picker 是日期字符串、multi_select 是数组
- form_value 各组件类型：
  - checker：`{name: true/false}`
  - date_picker：`{name: "2026-07-24 +0800"}`（含时区）
  - multi_select：`{name: ["v1","v2"]}`
  - input：`{name: "用户输入文本"}`
- Service 解析时按类型取。

## V8. form_submit 按钮靠 name 路由（非 action_id）
- form_submit 按钮无 value/action_id，Service 必须按 `event.action.name` 路由（如 btn_pass/btn_reject/confirm_btn/btn_activate/btn_defer）。
- form 外按钮按 `action.value.action_id` 路由（如 story1_确认方案/story5_确认日终总结/story6_已阅/story8 全不选）。
- **两类按钮路由策略不同**，feishu_card.py 要分别处理。

## V9. 客户端版本要求
- schema 2.0 需飞书客户端 **7.20+**。低于此版本卡片内容显示兜底"请升级"提示（标题仍显示）。
- 本项目实测 form/checker/date_picker 全部正常工作，客户端满足要求。
