## S1 目标规划与确认

无 form，概览 + 确认按钮。

**确认前（初始推送）**
```
📋 方案总览                        [蓝]
目标：知识库构建
专题数：4  阶段数：12  任务数：36
确认后将正式建库，可在 H5 页面调整。
                  [确认方案]
```
- 蓝色标题（待操作）
- 概览只展示数量，不展示全部细节
- 「确认方案」按钮回传 `action_id=story1_确认方案` + `draft_id`

**点击「确认方案」后** -> Service 读 draft -> 写 goals/themes/phases/tasks -> 删 draft -> update_card
```
📋 方案总览                        [绿]
目标：知识库构建
专题数：4 / 阶段数：12 / 任务数：36
✅ 方案已确认，已正式建库
[前往配置页调整]
```
- 标题转绿（已完成），按钮移除
- 「前往配置页调整」文字链接跳 H5

## S2 调度激活（含项目空间）

patch A->B 两态 + 确认后态。checker 勾选专题 + date_picker 选 deadline。

**状态1：卡片 A（选专题）**
```
🎯 调度激活                        [蓝]
目标：知识库构建
请勾选要激活的专题：
☑ 知识获取（learning）
☑ 知识沉淀（learning）
☐ 知识库架构RAG（learning）
☐ 知识管理闭环（learning）
                  [下一步]
默认全托管，调整 managed/path 请前往配置页
```
- 每个专题一个 checker（name=`theme_<id>`）
- 「下一步」form_submit 按钮（name=`next_btn`，不能带 behaviors）
- 配置页链接文字（managed 默认托管，调整走配置页）

**点击「下一步」后** -> 按 `next_btn` 识别 -> 取勾选专题 -> 锁定每个的第 1 个未开始阶段（强约束）-> update_card patch 为卡片 B

**状态2：卡片 B（填 deadline）**
```
🎯 调度激活 - 填 deadline          [蓝]
目标：知识库构建
以下阶段将被激活，请填 deadline：
· 知识获取 / 阶段1：知识获取（learning）  deadline：[选择日期]
· 知识沉淀 / 阶段1：知识沉淀（learning）  deadline：[选择日期]
                  [确认调度]
```
- 每个勾选专题：阶段名 + date_picker（name=`dl_<theme_id>`，required）
- 「确认调度」form_submit 按钮（name=`confirm_btn`）

**点击「确认调度」后** -> 按 `confirm_btn` 识别 -> 取各 date_picker 值 -> 校验全局进行中+本次 ≤3 -> 事务 UPDATE phases（进行中, activated_at, deadline）+ 即时级联 + status_change_log -> 异步工作空间初始化（managed=1）-> update_card

**状态3：已确认**
```
🎯 调度已确认                      [绿]
目标：知识库构建
✅ 调度已确认，已激活以下阶段：
· 知识获取 / 阶段1 - deadline 2026-07-15
· 知识沉淀 / 阶段1 - deadline 2026-07-18
工作空间正在初始化。
```
- 绿色标题，按钮全移除
- 显示激活阶段 + deadline + 初始化进度
- **确认后 managed 已锁定，不提供配置页链接**；调整须在确认前走配置页

## S3 当日计划推送

form + checker（候选任务 + 前置，两组独立勾选）。

**状态1：今日计划卡（初始推送）**
```
📋 今日计划（2026-07-02）          [蓝]
候选任务（勾选今日要做的）：
☑ 理论推导：信息获取渠道设计（知识获取/阶段1）[人]
☑ 代码实现与工程化：部署 RSSHub（知识获取/阶段1）[人]
☐ 总结+面试题库（知识获取/阶段1）[人]
今日前置（可取消）：
☑ 准备 RSSHub 环境
☐ 阅读 SimHash 原理
                  [确认今日计划]
```
- 候选任务 checker（name=`task_<id>`，text 含任务名/阶段/[executor]）
- 前置独立一组 checker（name=`pre_<id>`），与任务解耦
- 「确认今日计划」form_submit（name=`confirm_btn`）

**点击「确认今日计划」后** -> 按 `confirm_btn` 识别 -> 取勾选任务 + 勾选前置 -> 事务 INSERT daily_records + daily_tasks + subtasks -> 异步 opencode 执行前置 + 启动 agent 任务 -> update_card

**状态2：已确认**
```
📋 今日计划已确认（2026-07-02）    [绿]
✅ 今日计划已确认
今日任务：
· 理论推导：信息获取渠道设计 [人]
· 代码实现与工程化：部署 RSSHub [人]
今日前置：
· 阅读 SimHash 原理
前置与智能体任务已异步执行。
```
- 绿色标题，按钮移除，显示勾选任务 + 前置 + 异步执行提示

## S4A 智能体执行验收

form + 两个 submit（验收通过/需要修改）+ input feedback；场景3纯提醒无按钮；场景4确认完成推卡。

**场景1：验收卡（初始推送，前 1-3 次）**
```
✅ 智能体产出验收                  [蓝]
任务：<task_name>
产出文件：
· <file1>
· <file2>

┌─ form ──────────────────────────────┐
│ [验收通过]                          │  <- btn_pass（primary）
│ ──────────────────────────────────  │
│ 若需修改，请填写反馈后点「需要修改」│
│ [输入修改建议............]          │  <- feedback input
│ [需要修改]                          │  <- btn_reject（danger）
└─────────────────────────────────────┘
```
- form 内顺序：验收通过（上）-> 分隔 -> feedback input -> 需要修改（下）
- 两 submit 按钮靠 name 区分（`btn_pass` / `btn_reject`），不能带 behaviors
- `feedback` input 收修改建议（仅"需要修改"时用）

**点击「验收通过」后** -> 按 `btn_pass` 识别 -> UPDATE tasks 已完成 + 即时级联 + status_change_log -> update_card
```
✅ 验收通过                        [绿]
任务：<task_name>
✅ 验收通过，任务已完成
产出文件：
· <file1>
· <file2>
```

**点击「需要修改」后（前 1-3 次）** -> 按 `btn_reject` 识别 -> 取 feedback -> retry_count+=1（<3）-> 反馈下发 opencode 重新执行 -> update_card
```
⚠️ 反馈已下发                      [橙]
任务：<task_name>
✅ 你的反馈已下发智能体，等待其调整产出
你的反馈：
> <feedback内容>
产出文件：
· <file1>
```
- 橙色标题；显示用户填的 feedback

**场景3：第 4 次（retry_count 达 3，纯提醒无按钮）** -> DELETE /session/:sessionID 退该 task session（全局 serve 保留）-> 推卡
```
需人工接手                        [红]
任务：<task_name>
智能体 3 次验收未通过，需你人工接手处理。
请在本地工作空间用以下会话接管该任务，完成后发送 /pm 确认完成 让系统重新驱动后续智能体任务。
────────────────────────
项目空间路径：<workspace_path>
会话 ID：<session_id>
────────────────────────
提示：全局 opencode serve 进程持续运行（服务其他任务），你只需用上述 session_id 在本地接管本任务。
```
- 红色标题，**无任何按钮**（确认完成走 `/pm 确认完成` 聊天指令）
- 给项目空间路径 + session_id

**场景4：/pm 确认完成 推卡**（用户发指令，不带任务名则推卡）
```
确认完成任务                      [蓝]
工作空间：<workspace_name>
请确认以下任务的完成情况。
✅ 已完成任务：
· <task_name> [人]
· <task_name> [智能体]
⏳ 待确认完成（勾选后点确认）：
☑ <task_name> [人]
☑ <task_name> [智能体]
                  [确认完成]
```
- 已完成任务只读展示（含 [人]/[智能体]）
- 待确认任务 checker 勾选确认完成（name=`task_<id>`）
- 「确认完成」form_submit（name=`confirm_btn`）

**点击「确认完成」后** -> 按 `confirm_btn` 识别 -> 勾选 task 标记完成 + 即时级联 -> update_card
```
确认完成已提交                    [绿]
✅ 确认完成已提交
工作空间：<workspace_name>
本批确认完成的任务：
· <task_name> [人] - 已完成
· <task_name> [智能体] - 已完成
系统已记录。
```

## S4B 人完成任务（后置确认 + 完成通知）

form + checker（后置勾选）+ 全选/全不选小按钮 + 确认；完成后非阻塞通知卡。

**状态1：任务完成确认卡（初始推送）**
```
✅ 任务已完成                      [蓝]
任务：<task_name>
任务已完成。请确认后置收尾项（取消不需要的，可全取消）：
☑ 归档产出文件到知识库
☑ 更新面试题库
☑ 生成自测题
                [全选]  [全不选]
                  [确认]
```
- 后置 checker 默认全选（可取消）
- 全选/全不选小按钮在确认上方（form 外 behaviors callback，`story4B_全选` / `story4B_全不选`），点了 update_card 刷新所有 checker
- 只有一个「确认」form_submit（name=`confirm_btn`），无"不需要后置"按钮（全不选+确认 = 无后置）
- 后置属单个任务（带任务名）

**点击「全不选」后** -> 收到回调 -> update_card 把所有 checker checked=false（不提交 form）；全选同理

**点击「确认」后** -> 按 `confirm_btn` 识别 -> 取勾选后置 -> INSERT subtasks（全 false 则不插入）-> 任务保持已完成 -> update_card

**状态2：确认后中间态**
```
✅ 任务已完成                      [绿]
任务：<task_name>
✅ 后置已确认（无后置收尾）
任务保持已完成状态，无后置子任务执行。
```
（有勾选场景显示"后置工作已启动，完成后通知你"）

**状态3：后置执行完成通知卡**（异步执行完后推，非阻塞无按钮）
```
✅ 后置工作已完成                  [绿]
任务：<task_name>
✅ 后置工作已完成
执行的后置项：
· <post1>
· <post2>
产出文件：
· <file>
```
- 绿色标题，带任务名，非阻塞无按钮
- 全不选场景不推此卡

## S8 阶段衔接卡

form + date_picker（改 deadline）+ 确认激活/暂不激活水平并列。

**状态1：衔接卡（阶段完成事件触发推送）**
```
🎯 阶段衔接                        [蓝]
✅ 阶段「<completed_phase>」已完成
下一阶段：<next_phase>
请确认 deadline 并激活：
deadline：[选择日期（建议 <suggested_deadline>）]
      [确认激活]   [暂不激活]
```
- date_picker 确认/改 deadline（initial_date 是 Supervisor 建议值）
- 确认激活/暂不激活用 column_set 水平并列（form_submit，`btn_activate` / `btn_defer`）

**点击「确认激活」后** -> 按 `btn_activate` 识别 -> 取 deadline -> UPDATE phases（进行中, activated_at, deadline）+ 即时级联 + status_change_log -> 异步工作空间初始化 -> update_card
```
✅ 阶段已激活                      [绿]
✅ 下一阶段已激活：<next_phase>
· deadline：<deadline>
· 工作空间已就绪
阶段已进入进行中，即时级联已触发。
```

**点击「暂不激活」后** -> 按 `btn_defer` 识别 -> 记录暂缓（Redis 去重，24h 后 scheduler 再提醒）-> update_card
```
⏳ 已记录暂缓                      [橙]
下一阶段：<next_phase>
⏳ 已记录暂缓激活，24 小时后将再次提醒
如需立即激活，前往 H5 页面手动操作。
```

## S8 其他子卡（按通用规则实现，无新交互模式）

**专题完成卡**（专题所有阶段完成 -> Supervisor on_theme_completed）
- 内容：专题X已完成 + 列出其他未完成专题供用户选
- 按钮：每未完成专题一个「激活此专题」按钮（form 外 behaviors callback）
- 专题无序，完成时不自动锁定下一专题，给清单让用户选

**目标完成卡**（目标所有专题完成 -> Supervisor on_goal_completed）
- 内容：目标X全部完成 🎉
- 纯通知，无按钮

**巡检提醒卡**（4 种，scheduler 定时，纯提醒 + 跳转按钮）
- 开始日未激活提醒：build_start_date_reminder_card
- deadline 临近提醒（前1天/当天）：build_deadline_reminder_card
- 未确认今日计划提醒（10:00）：build_plan_reminder_card
- 未做日终总结提醒（21:00）：build_summary_reminder_card
- 均为纯提醒 + 跳转按钮（去激活/去 H5），form 外 behaviors callback；每项每天最多 1 次（Redis 去重）

## S7 子任务配置（无飞书卡片，走 H5）

配置走 H5 页面（子任务模板板块 CRUD：POST/PUT/DELETE /api/v1/subtask-templates）。
- 配置粒度：专题/阶段；只对 executor=人 的任务应用；阶段级优先于专题级；同名去重
- 不校验专题 type；去掉文件上传/下载
- 配置保存立即生效，下次生成计划时自动应用
- 无飞书卡片（纯 H5 CRUD，不推卡通知）

---

## S9 轻量编辑与状态回退（无飞书卡片，走 H5）

所有调整走 H5 页面调 Service API -> 落库。
- 字段编辑：名称/描述/deadline/executor
- 增删任务（已激活阶段下，物理删除，非 inactive 标记）
- 阶段排序（拖拽，任务排序不支持）
- 状态变更：暂停（进行中->已暂停，填 reason）/恢复（不填 reason）/回退（已完成->待执行，填 reason，触发即时重算级联）
- 项目空间 managed/path 只读查看（设置在 S2 卡片A，激活后不能改 managed）
- 无飞书卡片（H5 操作结果页面内反馈，不推卡通知）
