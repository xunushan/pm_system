# 一、背景

## 1.1 改造起因

v1.x 版本系统存在以下问题，用户在 2026-07-07 提出流程优化需求：

| # | 问题 | 证据 |
|---|------|------|
| 1 | 交互多（规划反复改卡片） | 01 用户故事 v1.3 Story 0+1 主流程：用户靠聊天逐条改（"加一个 Agent 框架""MLP 加 Dropout""算法题增加二叉树"） |
| 2 | 主动驱动偏多 | 05 Skill 设计 v1.2 路由表全是用户指令驱动，系统无主动巡检 |
| 3 | 级联复杂且滞后 | 02 数据模型 v1.4 第 2.14 节："触发时机：仅在确认日终总结时执行级联检查" |
| 4 | 阶段衔接断裂 | 阶段完成后无自动衔接，下阶段需重走 Story 2 全流程（确认目标→确认日期→确认阶段→确认 deadline→确认调度） |
| 5 | 无轻量编辑 | 已激活实体只有日终单条 PATCH（06 操作流程 v1.2 F 表步骤 4），无 CRUD 通道；pm-plan 的"轻量编辑"只在草稿态 |

## 1.2 用户提出的修改原则

1. **降低对话交互复杂度**：必要的操作跳转到页面上。
2. **Skill 聚焦效果而非应答式交互流程**。
3. **PM 助手轮询状态，更好地掌握完成情况，更高效地做出反馈，避免流程断裂**。

五条问题根因指向同一结构缺陷：**缺少常驻的"状态监听+编排"层，且结构化操作与对话操作没有分离**。

# 二、原则判断

## 2.1 原则 1「降对话、转页面」—— 方向正确，载体需界定

**合理**：当前把结构化编辑（增删改专题/阶段/任务字段）硬塞进"自然语言 → LLM 理解意图 → patch drafts → 刷卡片"链路，本质是用对话干表单的活。表单类操作天然适合页面 CRUD，对话只适合开放讨论和确认。

**风险与决策**：现有架构是"飞书 Bot + 卡片回调"，无 Web 前端。引入"页面"是架构级变更，形态决定成本。经确认采用**飞书多维表格**作为状态看板（见决策 D1），避免新建 Web App 的成本。

## 2.2 原则 2「Skill 聚焦效果而非应答式交互流程」—— 切中要害

**合理**：原 Skill 大量篇幅描述"Step1…StepN 的对话剧本"，把交互流程硬编码进 Skill，导致每个能力绑死一条对话链，难以被定时任务或巡检逻辑复用。

**决策**：Skill 重构为"原子能力 + 输入输出契约"，编排（谁调、何时调）交给上层调度器。同一能力可被用户指令、定时、巡检、回调四类入口复用（见决策 D3）。

## 2.3 原则 3「轮询状态、主动反馈」—— 升级为「事件即时 + 定时巡检兜底」

**合理**：纯事件驱动（用户指令 + 8:30/21:00/周日三个定时点）导致状态变化无人持续监听，是断裂根源。

**修正决策**：纯轮询成本高、延迟大。采用"事件即时（任务完成回调 → 即时级联/衔接下一阶段）+ 定时巡检兜底（deadline 临近、任务卡住、日终未总结）"。同时解掉原则 3 和问题 3（级联滞后）（见决策 D2、D5）。

# 三、关键决策清单

## D1. 状态看板载体：飞书多维表格

| 项 | 内容 |
|---|------|
| 决策 | 采用飞书多维表格作为状态看板，承载专题/阶段/任务的 CRUD、状态可视化、轻量编辑 |
| 备选 | A 飞书多维表格；B 飞书交互式卡片；C 独立 Web App |
| 选择理由 | 复用飞书基建，与现有 Bot+卡片回调架构融合度最高；支持内联编辑、筛选、多视图；架构改动量最小。B 复杂编辑受限，C 需新建前端+鉴权改动最大 |
| 双向同步规则 | DB 为唯一真相源，多维表格为受控视图。表格编辑 → Service 校验落库；其它入口变更 → 反向同步表格；冲突以 DB 为准 |
| 待验证 | 飞书多维表格 record 变更事件订阅能力（字段级回调粒度、延迟、webhook 支持）需在实现前核对飞书开放平台 Bitable 事件订阅 API。若不足，降级为"表格只读展示 + 编辑入口走卡片/对话" |
| 影响 | 03 架构新增入口 C；02 新增 BoardAppSvc；04 新增看板编辑模块、多维表格回调接口 |

## D2. 级联时机：即时级联

| 项 | 内容 |
|---|------|
| 决策 | 级联触发时机从"仅在确认日终总结时"改为"任务/阶段状态变更时即时触发"（事务内级联检查） |
| 原方案问题 | 绑死日终总结（02 数据模型 v1.4 第 2.14 节），导致状态滞后，问题 3 根源 |
| 事务边界 | 级联检查为纯 DB 查询+UPDATE，与状态变更同事务（<200ms，满足飞书 3 秒回调）；副作用（工作空间初始化、后置子任务、衔接推送）事务提交后异步 |
| 连带调整 | 日终总结退化为纯回顾+快照（不再级联）；daily_records.is_confirmed 语义从"级联触发开关"变为"回顾确认标记" |
| 影响 | 02 第 2.15 节改写；04 tasks/complete、output/confirm、board/status 均含即时级联；daily/summary/confirm 移除级联 |

## D3. Skill 重构为 5 个原子能力

| 项 | 内容 |
|---|------|
| 决策 | Skill 从 6 个剧本式（pm/pm-plan/pm-schedule/pm-task/pm-workspace/pm-config）重构为 5 个原子能力（pm/pm-plan/pm-daily/pm-subtask/pm-summary） |
| 判定标准 | **唯一标准：能力的核心逻辑是否需要 LLM 推理/生成/理解**。需要→Skill；纯确定性逻辑（CRUD、查询、排序、状态机、级联、巡检、HTTP 调用）→ Service 代码 |
| 去剧本化 | Skill 以"能力 + 输入输出契约"描述，编排交由调度层（用户指令/定时/巡检/回调四类入口统一复用同一能力） |
| 下沉为代码的能力 | 调度候选查询、歧义澄清、预览卡片、任务定位、完成确认、级联、智能体下发/回调/验收卡片、模板 CRUD、巡检、阶段衔接推算、轻量编辑落库 |
| 数量控制 | 仅 5 个 Skill，每个仅承担 LLM 不可替代的核心逻辑，避免智能体上下文加载过多 Skill 描述 |
| 命名 | 沿用 `-` 分隔（pm-plan/pm-daily/pm-subtask/pm-summary） |
| 影响 | 05 Skill 设计 v2.0 重写；08 章 Skill 与 Service 能力边界对照 |

## D4. pm-daily 由 LLM 决策今日计划

| 项 | 内容 |
|---|------|
| 决策 | 今日计划由 LLM 综合决策（昨日完成+当前阶段进度+deadline 紧急度+任务依赖+精力），非纯 deadline 排序 |
| 起因 | 用户指出"当天的任务计划生成需要 LLM 基于昨天完成的和当前阶段下任务综合判断今天的安排" |
| 分工 | Service 代码：任务池筛选（捞取待执行、过滤 scheduled_start_date、过滤已暂停）；LLM Skill：综合决策今天做什么、做多少、怎么排 |
| 新增 API | GET /api/v1/daily/plans/pool（任务池预查询，供 LLM 输入） |
| 影响 | 05 新增 pm-daily Skill；04 新增 plans/pool 接口；01 Story 3 主流程改写 |

## D5. 主动反馈模型：事件即时 + 定时巡检兜底

| 项 | 内容 |
|---|------|
| 决策 | 新增 Supervisor 组件（Service 内常驻），事件即时 + 定时巡检兜底 |
| 原则修正 | 用户原文"轮询"，升级为"事件即时 + 定时巡检兜底"（纯轮询成本高延迟大） |
| 事件即时 | 阶段完成→衔接下一阶段；任务完成→即时级联；智能体产出→验收卡片 |
| 定时巡检兜底 | deadline 临近（前1天/当天）、任务卡顿（≥2天无变更）、未确认计划（10:00）、未做日终总结（21:00）、阶段衔接 24h 未响应 |
| 防骚扰 | 每项每天最多 1 次；已暂停实体不巡检；Redis 记录去重 key |
| Supervisor 不进 Skill | 巡检与衔接推算为纯确定性逻辑（查询+计算），无 LLM，为 Service 代码 |
| 影响 | 03 新增 Supervisor 组件；01 新增 Story 9/10；06 新增 J/K 操作流程 |

## D6. 状态机扩充：已暂停态 + 有限回退

| 项 | 内容 |
|---|------|
| 决策 | 阶段/任务/专题/目标新增"已暂停"态；支持有限回退（已完成→进行中/待执行） |
| 起因 | 支持轻量编辑/纠错（问题 5），原状态机单向流转不可回退 |
| 流转规则 | forward（正常）、pause（进行中→已暂停）、resume（已暂停→进行中）、revert（已完成→进行中/待执行，reason 必填） |
| 回退规则 | revert 必填 reason；触发即时重算级联（可能把已完成的上级拉回进行中）；所有变更写 status_change_log |
| 已暂停语义 | 不纳入每日计划推荐、不推 deadline 提醒、不占全局并发名额 |
| 连带调整 | 放宽 CHECK 约束（02 数据模型 v1.4 第 2.13 节单向流转）；主表不冗余审计字段，新增 status_change_log 表 |
| 影响 | 02 新增 status_change_log 表、放宽 CHECK；03 第 8.8 节状态机约束；04 board/status 接口、错误码 1005 |

## D7. 项目空间托管模式：managed 字段

| 项 | 内容 |
|---|------|
| 决策 | workspaces 新增 managed 字段。managed=1（默认）系统托管，激活时初始化（mkdir+git init+骨架）；managed=0 关联已有路径，激活时跳过初始化 |
| 起因 | 用户新需求"希望能够设置专题关联的项目空间，设置好项目空间的专题不需要进行初始化" |
| 校验 | managed=0 时 Service 校验 path 存在性，不碰目录结构/git，直接置"已就绪"；系统仍可往该路径读写文件 |
| 设置入口 | 属配置类 CRUD，不建 Skill——看板直接编辑（填 path + managed=0）或对话经 pm 主路由 LLM 提取参数后调 Service |
| 主 Agent 进程不受影响 | opencode serve 基于 workspace.path 启动，托管/关联一致 |
| 新增 API | PUT /api/v1/workspaces/{id}/link |
| 影响 | 02 workspaces 表新增 managed；03 第 8.11 节工作空间初始化幂等性；04 workspaces/link 接口；01 新增 Story 8 |

## D8. 配置类操作不建 Skill

| 项 | 内容 |
|---|------|
| 决策 | 模板配置、项目空间关联设置为纯字段 CRUD，走 ConfigAppSvc/WorkspaceAppSvc 代码，不建独立 Skill |
| 依据 | 纯确定性逻辑（D3 判定标准） |
| 对话路径 | 仅用于需 LLM 理解意图的场景（如自然语言描述变更意图）；pm 主路由 LLM 提取参数后直接调 Service |

| 影响 | 05 Skill 清单不含 pm-config（原 v1.2 有）；04 配置类 API 标注为代码能力 |

## 8.5 飞书卡片能力调研结论（支撑 D9、D18）

**支持的交互组件**：
- button（按钮回调，带 action_value）
- multi_select_static（静态多选）
- select_static（静态单选）
- date_picker（日期选择）

- form（表单容器，组合提交）

- 级联选择（cascader，A 选了 B 动态变）—— 选项静态预定义，无法交互时动态请求后端刷新

- 卡片适合"选项已确定的单次确认"（勾选、输入、单选、日期）

# 十、第四轮 review 决策（v1.3 追加，2026-07-08）

review Story 3 和 Story 4B 时，发现前置/后置的生成机制和勾选逻辑需要厘清，涉及 LLM 调用边界硬约束。本章节记录第四轮决策。

## 10.1 发现的问题

### 问题 1：前置和任务勾选的联动，Service 处理不了

**问题**：前置由 LLM 生成（pm-subtask）。如果用户在今日计划卡片上改了任务勾选（取消一个任务），前置要不要跟着变？变动需要 LLM 重新生成前置，但 Service 不调 LLM（架构原则），处理不了。

**根因**：任务和前置若有联动，就需要 LLM 重新生成，与"Service 不调 LLM"冲突。

### 问题 2：后置和任务完成的关系不清晰

**问题**：后置是"任务完成后的收尾"。但任务完成（标记已完成）和后置执行是绑定的还是脱钩的？之前 Story 4B 定的"确认完成才触发后置"，意味着用户不做后置，任务就一直挂着"待执行"，但实际已做完——不合理。

## 10.2 新增决策

### D22. 前置按今日整体生成 + 与任务解耦

| 项 | 内容 |
|---|------|
| 决策 | 前置按"今日整体"生成（不按单个任务），与任务勾选解耦 |
| 起因 | 任务和前置联动需要 LLM 重新生成，Service 不调 LLM，处理不了 |
| 生成时机 | pm-daily 生成今日计划时，调 pm-subtask 生成"今日整体前置"（一次 LLM 调用） |
| 卡片交互 | 候选任务勾选 + 前置勾选，两组独立，互不影响。取消任务不影响前置 |
| 用户操作 | 用户只能勾选/取消候选任务（不能加新任务），前置独立勾选可取消 |
| 前置归属 | 不区分任务归属，就是"今日整体的准备" |
| 影响 | 01/06 Story 3 改写 |

### D23. 后置和任务完成脱钩

| 项 | 内容 |
|---|------|
| 决策 | 任务完成和后置脱钩：pm-subtask 先调 Service 标记完成（即时级联），再生成后置清单。任务在标记完成时就已完成，不等后置确认 |

# 十一、第五轮 review 决策（v1.4 追加，2026-07-08）

review Story 1 时，用户提出"目标拆解数据较大，飞书回调能否处理这么大数据"。本章节记录决策。

## 11.1 发现的问题

### 问题：规划数据量大与飞书回调数据量限制的冲突

**问题**：一个目标的完整规划（目标+4专题+24阶段+72任务）数据量可达几十 KB（约 30-60KB）。飞书卡片回调（card.action.trigger）请求体上限约 30KB。如果确认按钮回调携带完整规划数据，可能超限失败。

**关联问题**：之前 v1.1 决策 D10 废弃了 drafts 表（理由是"草案进看板"复杂）。但废弃 drafts 后，pm-plan 生成的规划数据在确认前无处存储（Service 收到回调时拿不到 Skill 内存数据）。

## 11.2 新增决策

### D24. drafts 表恢复用于确认前存储（修正 D10）

| 项 | 内容 |
|---|------|
| 决策 | 恢复 drafts 表，用于 pm-plan 生成规划数据的确认前存储（纯存储，不同步展示） |
| 修正 | D10（废弃 drafts）——区分"草案进看板展示"（仍废弃）与"确认前数据存储"（恢复） |
| 起因 | 飞书回调约 30KB 限制，规划数据可达几十 KB；且 Service 收到回调时拿不到 Skill 内存数据 |
| 数据流 | pm-plan 生成规划 → 调 Service 写 drafts（content 存完整规划 JSON）→ 总览卡片按钮 action_value 只含 draft_id → 确认回调只传 draft_id → Service 用 draft_id 读 drafts → 写正式表 → 删 drafts |

## 11.3 飞书卡片数据量原则（通用）

**原则**：所有飞书卡片设计避免一次性传大数据。卡片只展示概览或少量项，细节在 H5 页面。回调只传标识符（draft_id、task_id 等），不传完整数据。

**各卡片数据量评估**：
| 卡片 | 数据量 | 是否超限 |
|---|---|---|
| Story 1 总览卡片 | 概览（专题数/阶段数/任务数）+ draft_id | 否 |
| Story 2 卡片 A/B | 多选专题 + 阶段 + deadline | 否（专题≤几个，阶段≤3） |
| Story 3 今日计划 | 5-8 候选任务 + 前置 | 否 |
| Story 4A 验收 | 任务名 + 产出文件名列表 | 否（产出文件单独发） |
| Story 4B 后置勾选 | 后置清单 3-5 项 | 否 |
| Story 5 日终总结 | 今日任务列表 + 状态按钮 | 否（今日任务≤几个） |
| Story 6 周总结 | 每日回顾 + 健康度 | 否 |
| Story 8 衔接 | 下一阶段 + deadline | 否 |

**结论**：所有卡片均不超限。唯一可能超限的"完整规划数据"通过 drafts 表 + draft_id 传递，规避回调限制。
