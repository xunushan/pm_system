# 目标管理系统 - 数据模型文档

# 一、概念模型

## 1.1 概念清单

| 概念 | 定义 | 用户可见术语 |
|------|------|-------------|
| **目标** | 用户要达成的最终状态 | 目标 |
| **专题** | 目标下的独立推进板块（无序并列） | 方向 / 板块 |
| **阶段** | 专题内的可验收推进节点，按 roadmap 顺序推进（强约束） | 步骤 |
| **任务** | 阶段内的可执行、可验收的事项 | 具体要做的事 |
| **子任务** | 任务的拆分单元（前置/后置，只服务人执行任务） | （内部概念，不展示） |
| **工作空间** | 与专题 1:1 绑定的物理命名空间 | 项目空间 |
| **项目空间托管模式** | managed=1 系统托管（激活初始化）；managed=0 关联已有路径（跳过初始化，不创建任何文件） | （内部概念，不展示） |
| **每日计划** | 当天所有任务的汇总视图 | 今日计划 |
| **每日任务** | 当天计划执行的具体任务条目 | 今日任务 |
| **草稿** | pm-plan 确认前的规划数据存储（纯存储，不同步展示） | （内部概念，不展示） |
| **子任务模板** | 用户配置的固定前置/后置工作描述 | （内部概念，不展示） |
| **状态变更日志** | 实体状态变更的审计记录（含回退/暂停原因） | （内部概念，不展示） |

## 1.2 实体关系

```
目标 (Goal) [1]
  └── 专题 (Theme) [N]  ← 无序并列
       ├── 阶段 (Phase) [N]  ← roadmap 强约束，按 sort_order 顺序激活
       │    └── 任务 (Task) [N]  ← executor 规划态不填，pm-daily 推断
       │         └── 子任务 (Subtask) [N]  ← 前置/后置，只服务人执行任务，智能体执行
       └── 工作空间 (Workspace) [1:1]
              └── managed=1 系统托管 | managed=0 关联已有路径

子任务模板 (Subtask Template) [N]  ← 与专题或阶段关联
  ├── scope_type='theme' → 关联专题
  └── scope_type='phase' → 关联阶段

每日计划 (Daily Record) [1]
  └── 每日任务 (Daily Task) [N]
       └── 引用 → 任务 (Task)

草稿 (Draft) [N]  ← pm-plan 确认前存储规划数据（纯存储）
状态变更日志 (Status Change Log) [N]  ← 审计所有实体状态变更
```

## 1.3 关键约束（实体层面）

| # | 约束 | 说明 |
|---|------|------|
| 1 | 专题 ↔ 工作空间 **严格 1:1** | 一个专题对应一个工作空间；managed=1 系统托管（激活初始化），managed=0 关联已有路径（跳过初始化，不创建任何文件） |
| 2 | 专题 **无序** | 专题并列无序，跨专题可并行推进；专题无 sort_order |
| 3 | 阶段 **roadmap 强约束** | 专题内阶段按 sort_order 顺序激活，不可跳；激活时自动锁定第1个未开始阶段 |
| 4 | 跨专题 **可并行推进** | 全局"进行中"阶段 ≤ 3（"已暂停"不占名额） |
| 5 | 任务可拆分子任务 | 前置/后置只服务人执行任务，智能体执行 |
| 6 | 状态唯一真相源 | 任务状态以 tasks 表为准，daily_tasks 为快照 |
| 7 | 子任务模板无顺序性 | 模板不维护 sort_order，查询时去重合并 |
| 8 | 模板作用域优先级 | 阶段级模板优先于专题级，同名去重 |
| 9 | 级联即时触发 | 任务/阶段状态变更时即时向上级联，不依赖日终总结 |
| 10 | 状态变更可审计 | 所有状态流转写入 status_change_log，回退/暂停必须记录原因 |
| 11 | executor 规划态不填 | pm-daily 按专题 type 推断（learning/research/source→人；dev/survey→智能体） |
| 12 | scheduled_start_date 用于提醒激活 | 规划态确认（计划开始日），到点未激活提醒；每日计划改用 phases.activated_at 过滤 |

## 1.4 专题/阶段拆解原则

**优先级**：原则1 > 原则2 > 原则3 > 原则4

> **重要前提**：专题按**独立推进单元**划分，而非代码模块。判断标准：该单元能否**独立启动、独立验收、独立关闭**？

### 原则1：并行独立性（最强）
若两个单元需要同时推进，且各自有独立的工作空间需求，则拆为专题。

### 原则2：环境边界
同一专题内的所有阶段必须共享同一套技术环境。

### 原则3：工作模式一致性
同一专题内的所有阶段必须属于同一工作模式（学习 / 工程 / 调研 / 研究）。

### 原则4：认知域一致性
同一专题内的所有阶段必须围绕同一核心问题空间。

### 专题拆解流程

```
Step 1: 并行独立性测试 → 是 → 新建专题
Step 2: 环境边界测试 → 否 → 新建专题
Step 3: 工作模式测试 → 否 → 新建专题
Step 4: 认知域测试 → 否 → 新建专题；是 → 归入现有专题作为新阶段
```

## 2.2 核心规划表

```sql
-- 1. 目标表
CREATE TABLE goals (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL UNIQUE,
    description TEXT,
    time_range_start DATE,  -- 目标时间范围（粗略，规划态确认）
    time_range_end DATE,
    scheduled_start_date DATE,  -- 计划开始日（规划态确认，用于提醒激活；每日计划不用此过滤）
    status TEXT CHECK(status IN ('未开始','进行中','已完成','已暂停')) DEFAULT '未开始',
    status_changed_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 2. 专题表（无序，去 sort_order/time_range）
CREATE TABLE themes (
    id TEXT PRIMARY KEY,
    goal_id TEXT NOT NULL REFERENCES goals(id),
    name TEXT NOT NULL,
    description TEXT,
    type TEXT CHECK(type IN ('learning','research','source','dev','survey')) DEFAULT 'learning',  -- 用于推断 executor
    status TEXT CHECK(status IN ('未开始','进行中','已完成','已暂停')) DEFAULT '未开始',
    status_changed_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 3. 阶段表（roadmap 强约束，保留 sort_order）
CREATE TABLE phases (
    id TEXT PRIMARY KEY,
    theme_id TEXT NOT NULL REFERENCES themes(id),
    sort_order INTEGER NOT NULL,  -- roadmap 顺序，按此激活
    name TEXT NOT NULL,
    description TEXT,
    status TEXT CHECK(status IN ('未开始','进行中','已完成','已暂停')) DEFAULT '未开始',
    status_changed_at TIMESTAMP,
    deadline DATE,  -- 激活时填（Story 2 卡片 B）
    activated_at DATE,  -- 激活时间（每日计划以此过滤，不用 scheduled_start_date）
    completed_at TIMESTAMP,
    paused_at TIMESTAMP,
    UNIQUE(theme_id, sort_order)
);
```

> **专题无序**：themes 去 sort_order/time_range_start/time_range_end。专题并列无序，跨专题可并行推进。
> **阶段强约束**：phases 保留 sort_order，按 roadmap 顺序激活，激活时自动锁定第1个未开始阶段。

## 2.3 任务表（状态唯一真相源）

```sql
-- 4. 任务表
CREATE TABLE tasks (
    id TEXT PRIMARY KEY,
    phase_id TEXT NOT NULL REFERENCES phases(id),
    sort_order INTEGER NOT NULL,  -- 任务排序不强制（pm-daily 动态定），仅记录
    name TEXT NOT NULL,
    description TEXT,
    status TEXT CHECK(status IN ('待执行','已完成','已暂停')) DEFAULT '待执行',  -- 去"需人工介入"
    status_changed_at TIMESTAMP,
    executor TEXT CHECK(executor IN ('human','agent')),  -- 规划态不填（NULL），pm-daily 按专题 type 推断
    has_subtask BOOLEAN DEFAULT 0,
    retry_count INTEGER DEFAULT 0,
    completed_at TIMESTAMP,
    paused_at TIMESTAMP,
    UNIQUE(phase_id, sort_order)
);
```

> **executor 规划态不填**：规划时 executor 为 NULL，pm-daily 推今日任务时按所属专题 type 推断填入（learning/research/source→human；dev/survey→agent）。用户可在 H5 页面改。
> **状态去"需人工介入"**：智能体任务 3 次重试不通过不改状态（保持原状态），用户介入后"/pm 确认完成"直接标记完成。
> **任务排序不强制**：sort_order 仅记录，pm-daily 动态决定今日任务顺序，H5 页面不支持任务排序调整。

## 2.4 工作空间表

```sql
-- 5. 工作空间表
CREATE TABLE workspaces (
    id TEXT PRIMARY KEY,
    theme_id TEXT NOT NULL UNIQUE REFERENCES themes(id),
    path TEXT NOT NULL,
    managed BOOLEAN DEFAULT 1,  -- 1=系统托管(激活初始化)，0=关联已有路径(跳过初始化，不创建任何文件)
    status TEXT CHECK(status IN ('未初始化','已就绪','运行中','空闲')) DEFAULT '未初始化',
    type TEXT CHECK(type IN ('learning','research','source','dev','survey')) DEFAULT 'learning',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    last_heartbeat TIMESTAMP
);
```

> **managed 字段语义**：
> - `managed=1`（默认，系统托管）：激活阶段时初始化（mkdir + git init + 骨架文件含规范文件），path 由系统生成。
> - `managed=0`（关联已有）：path 为用户指定路径，激活时**跳过初始化，不创建任何文件**（包括规范文件），仅校验 path 存在性后置已就绪。用户自己保证目录有规范文件。
> - managed/path 在 Story 2 卡片 A 设置（激活前最后一刻）；激活后不能改 managed（避免破坏已初始化目录）。

## 2.5 子任务表

```sql
-- 6. 子任务表（前置/后置，只服务人执行任务）
CREATE TABLE subtasks (
    id TEXT PRIMARY KEY,
    task_id TEXT NOT NULL REFERENCES tasks(id),
    sort_order INTEGER NOT NULL,
    name TEXT NOT NULL,
    description TEXT,
    type TEXT CHECK(type IN ('前置','后置')) NOT NULL,
    status TEXT CHECK(status IN ('待执行','进行中','已完成','失败')) DEFAULT '待执行',
    output_path TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    completed_at TIMESTAMP,
    UNIQUE(task_id, sort_order)
);
```

> **前置/后置只服务人执行任务**：智能体任务不生成前置/后置。前置按"今日整体"生成（不按单个任务），后置按单个任务生成。前置/后置都智能体执行（opencode run）。

## 2.6 子任务模板表

```sql
-- 7. 子任务模板表（用户配置）
CREATE TABLE subtask_templates (
    id TEXT PRIMARY KEY,
    scope_type TEXT CHECK(scope_type IN ('theme','phase')) NOT NULL,
    scope_id TEXT NOT NULL,
    type TEXT CHECK(type IN ('前置','后置')) NOT NULL,
    name TEXT NOT NULL,
    description TEXT,
    status TEXT CHECK(status IN ('active','inactive')) DEFAULT 'active',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(scope_id, type, name)
);
```

> **配置粒度**：专题或阶段，阶段级优先于专题级，同名去重。配置时不校验专题 type（智能体专题配了也不提示，生成时自然跳过）。

## 2.8 调度执行表

```sql
-- 9. 每日记录表
CREATE TABLE daily_records (
    id TEXT PRIMARY KEY,
    date DATE UNIQUE NOT NULL,
    week TEXT NOT NULL,
    push_source TEXT CHECK(push_source IN ('auto','manual')) DEFAULT 'manual',
    is_confirmed BOOLEAN DEFAULT 0,  -- 回顾确认标记（不触发级联）

CREATE TABLE daily_tasks (

-- 11. 每周记录表
CREATE TABLE weekly_records (
    id TEXT PRIMARY KEY,
    week TEXT UNIQUE NOT NULL,
    date_range_start DATE NOT NULL,

    is_confirmed BOOLEAN DEFAULT 0,
    confirmed_at TIMESTAMP,
    summary TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 2.9 草稿表（pm-plan 确认前存储规划数据）

```sql
-- 12. 草稿表（纯存储，不同步展示）
CREATE TABLE drafts (
    id TEXT PRIMARY KEY,
    user_id TEXT NOT NULL,
    story_type TEXT NOT NULL CHECK(story_type IN ('plan','schedule','daily','weekly','edit','config')),
    entity_id TEXT,
    content TEXT NOT NULL,  -- JSON 格式存储完整规划数据（含目标+专题+阶段+任务）
    status TEXT DEFAULT 'pending' CHECK(status IN ('pending','confirmed','expired','discarded')),
    version INTEGER DEFAULT 1,  -- 乐观锁
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    expires_at TIMESTAMP  -- 24h 过期
);
```

> **drafts 表用途（D24）**：pm-plan 生成规划时调 Service 写入 drafts（content 存完整规划 JSON，可达几十 KB）。确认按钮回调只传 draft_id（规避飞书回调约 30KB 限制）。Service 收到回调，用 draft_id 读 drafts → 写正式表 → 删 drafts。
> **纯存储**：drafts 不同步展示（不进 H5 页面/多维表格），只用于确认前数据传递。
> **逐专题追加**：pm-plan 逐专题生成，内容追加到同一个 draft。
> **乐观锁**：version 字段防止并发覆盖。

## 2.10 项目空间主 Agent 进程表

```sql
-- 13. 项目空间主 Agent 进程表
CREATE TABLE agent_processes (
    id TEXT PRIMARY KEY,
    workspace_id TEXT NOT NULL UNIQUE REFERENCES workspaces(id),
    port INTEGER NOT NULL,
    pid INTEGER,
    status TEXT CHECK(status IN ('running','crashed','stopped')) DEFAULT 'running',
    started_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    last_heartbeat TIMESTAMP,
    task_queue TEXT
);

CREATE INDEX idx_agent_processes_status ON agent_processes(status);
```

> **启动时机**：首次下发智能体任务时启动（Story 3 确认今日计划后），不是 Story 2 激活时。
> **生命周期**：阶段级常驻，阶段完成（即时级联触发）后退出。3 次重试不通过时退出，用户手动接管；"/pm 确认完成"后系统重新启动（不同端口）接管后续智能体任务。
> **不受 managed 影响**：基于 workspace.path 启动，托管/关联一致。

## 2.11 状态变更审计表

```sql
-- 14. 状态变更审计表
CREATE TABLE status_change_log (
    id TEXT PRIMARY KEY,
    entity_type TEXT NOT NULL CHECK(entity_type IN ('goal','theme','phase','task')),
    entity_id TEXT NOT NULL,
    from_status TEXT,
    to_status TEXT NOT NULL,
    change_type TEXT CHECK(change_type IN ('forward','pause','resume','revert','cascade')) NOT NULL,
    reason TEXT,  -- 回退/暂停必填
    triggered_by TEXT,  -- 'user'/'agent_callback'/'supervisor'/'cascade'
    changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_status_log_entity ON status_change_log(entity_type, entity_id, changed_at);
CREATE INDEX idx_status_log_type ON status_change_log(change_type, changed_at);
```

> **change_type 语义**：forward（正常正向）、pause（进行中→已暂停）、resume（已暂停→进行中）、revert（已完成→进行中/待执行，reason 必填）、cascade（级联自动）。

## 2.12 视图

```sql
-- 活跃阶段视图（已暂停不计入）
CREATE VIEW active_phases_view AS
SELECT p.id, p.name, p.deadline, p.activated_at, t.id AS theme_id, t.name AS theme_name, t.goal_id
FROM phases p JOIN themes t ON p.theme_id = t.id
WHERE p.status = '进行中';

-- 工作空间进展视图
CREATE VIEW workspace_progress_view AS
SELECT wp.workspace_id, wp.date, wp.task_id, t.name AS task_name,
       COUNT(wp.id) AS output_count, GROUP_CONCAT(wp.file_path) AS output_files
FROM workspace_progress wp JOIN tasks t ON wp.task_id = t.id
GROUP BY wp.workspace_id, wp.date, wp.task_id;

-- 今日任务视图
CREATE VIEW today_tasks_view AS
SELECT dr.date, dt.task_id, t.name AS task_name, t.status AS task_status,
       p.name AS phase_name, th.name AS theme_name, th.type AS theme_type, t.executor
FROM daily_records dr
JOIN daily_tasks dt ON dr.id = dt.daily_id
JOIN tasks t ON dt.task_id = t.id
JOIN phases p ON t.phase_id = p.id
JOIN themes th ON p.theme_id = th.id
WHERE dr.date = DATE('now');

-- 子任务模板合并视图（阶段优先于专题）
CREATE VIEW subtask_templates_merged_view AS
SELECT st.*, CASE WHEN st.scope_type = 'phase' THEN 1 ELSE 2 END AS priority
FROM subtask_templates st
WHERE st.status = 'active';
```

## 2.14 状态入口汇总

| 实体 | 状态枚举 | 唯一入口 | 说明 |
|------|----------|---------|------|
| 目标 | 未开始 / 进行中 / 已完成 / 已暂停 | goals.status | 级联自动更新 |
| 目标调度开始日期 | — | goals.scheduled_start_date | 规划态确认，提醒激活 |
| 专题 | 未开始 / 进行中 / 已完成 / 已暂停 | themes.status | 级联自动更新；无序 |
| 阶段 | 未开始 / 进行中 / 已完成 / 已暂停 | phases.status | roadmap 强约束；activated_at 用于每日计划过滤 |
| 任务 | 待执行 / 已完成 / 已暂停 | tasks.status | 去"需人工介入"；executor 规划态不填 |
| 工作空间 | 未初始化 / 已就绪 / 运行中 / 空闲 | workspaces.status | managed 分支 |
| 子任务 | 待执行 / 进行中 / 已完成 / 失败 | subtasks.status | 只服务人执行任务 |
| 子任务模板 | 生效中 / 已停用 | subtask_templates.status | 用户配置 |
| 草稿 | 待确认 / 已确认 / 已过期 / 已废弃 | drafts.status | 纯存储，24h 过期 |
| 主 Agent 进程 | 运行中 / 已崩溃 / 已停止 | agent_processes.status | 心跳检测 |

## 2.15 级联推导规则（即时化）

```
任务状态变更为 '已完成'（人确认 或 智能体回调）
       │
       ▼
【同步事务内即时级联检查】
  查该阶段所有任务：全 '已完成' → UPDATE phases.status='已完成'
       │
       ▼
  查该专题所有阶段：全 '已完成' → UPDATE themes.status='已完成'
       │
       ▼
  查该目标所有专题：全 '已完成' → UPDATE goals.status='已完成'
       │
       ▼
  发出"阶段/专题/目标完成"事件 → Supervisor 衔接
  所有级联变更写 status_change_log（change_type='cascade'）
```

**触发时机**：任务/阶段状态变更时即时触发（事务内，纯 DB，满足飞书 3 秒回调）。回退（已完成→进行中/待执行）同样即时重算级联。

**事务边界**：级联检查与状态变更同事务（<200ms）；副作用（工作空间初始化、后置执行、衔接推送）事务后异步。

## 2.16 状态机流转规则

### 阶段状态流转
```
未开始 ──forward──► 进行中 ──forward──► 已完成
                  │  ├──pause──► 已暂停 ──resume──► 进行中
                  └──revert──► 进行中（从已完成回退，reason 必填）
```

### 任务状态流转
```
待执行 ──forward──► 已完成
  ├──pause──► 已暂停 ──resume──► 待执行
  └──revert──► 待执行（从已完成回退，reason 必填）
```

### 回退规则
- `已完成→进行中/待执行`（revert）：必填 reason，触发即时重算级联（可能把上级拉回进行中）。
- `进行中→已暂停`（pause）：必填 reason；不纳入计划、不推 deadline 提醒、不占名额。
- `已暂停→进行中`（resume）：不填 reason。
- 所有状态变更写 status_change_log。

## 2.17 超时监控（Redis）

```
下发智能体任务时：SET task_timeout:{task_id} EX 7200（2小时）
验收通过/重试时：DEL task_timeout:{task_id}
Redis KeyExpirationEvent 触发：飞书通知"智能体执行超时"
```

## 2.18 子任务模板合并规则

```
查询条件：task_id → phase_id → theme_id
Step 1: 查询阶段级模板（scope_type='phase'）
Step 2: 查询专题级模板（scope_type='theme'）
Step 3: 合并去重（阶段级优先，同名以阶段级为准）
返回合并后模板列表（无顺序）
```

# 三、关键数据流

## 3.1 规划态数据流（含 drafts 存储）

```
用户聊天 → pm-plan LLM 生成四层结构
  → 逐专题生成，追加到 drafts（调 Service 写 drafts.content）
  → 发总览卡片（只展示概览）+ 确认按钮（action_value 只含 draft_id）
  → 用户点确认 → 回调只传 draft_id
  → Service 用 draft_id 读 drafts → 事务写入 goals/themes/phases/tasks → 删 drafts
  → 返回 H5 页面链接
```

> **drafts 纯存储**：确认前规划数据存 drafts，不进正式表（无 is_draft）。drafts 不同步展示。

## 3.2 调度态数据流（含 managed 分支 + 多选）

```
用户多选专题 + 设各专题 managed/path（Story 2 卡片 A）
  → patch 卡片 B（填各阶段 deadline）
  → 用户确认调度
  → 事务：UPDATE phases（进行中, activated_at=NOW(), deadline）+ 即时级联 themes/goals + status_change_log
  → 异步工作空间初始化（managed 分支）：
      managed=1 → mkdir + git init + 骨架含规范文件
      managed=0 → 校验 path，不创建任何文件，直接置已就绪
```

> **每日计划过滤**：用 phases.activated_at（激活即开始推送），不用 goals.scheduled_start_date。

## 3.3 执行态数据流（智能体，Story 4A）

```
确认今日计划（含 executor=智能体 任务）→ 异步启动 opencode serve → 下发任务
  → 智能体执行（按项目空间规范写产出文件）→ 回调
  → 记录 workspace_progress → 发验收卡片 + 发送产出文件到飞书
  → 用户验收通过 → UPDATE tasks 已完成 → 即时级联 → 发阶段完成事件 → Supervisor 衔接
  → 3 次不通过：系统进程退出，用户手动接管 → "/pm 确认完成" → 系统重新启动 opencode serve 接管后续
```

## 3.4 执行态数据流（人，Story 4B）

```
用户 "/pm 完成 [任务]" → pm-subtask Skill
  → 调 Service 标记完成 + 即时级联（任务此时已完成，脱钩）
  → pm-subtask 生成后置清单（LLM，只对人执行任务）
  → 后置勾选卡片 → 用户勾选（可全取消）→ 确认后置
  → 异步 opencode run 执行后置（智能体执行）
```

## 3.5 今日计划数据流（Story 3，前置整体生成）

```
8:30 / "今日计划" → pm-daily Skill
  → Service 任务池预查询（已激活阶段 + 排除已暂停）
  → pm-daily LLM 决策候选任务 + 推断 executor（按专题 type）
  → pm-daily 调 pm-subtask 生成"今日整体前置"（不按单个任务，与任务解耦）
  → 卡片：候选任务勾选 + 前置勾选（两组独立）
  → 用户勾选 → 确认 → INSERT daily_records/daily_tasks/subtasks（前置）→ 异步执行前置
```

## 3.6 配置态数据流（Story 7）

```
用户在 H5 页面"子任务模板"板块 CRUD → Service ConfigAppSvc 落库
  → 立即生效，下次生成计划时自动应用（只对 executor=人 的任务）
```

## 3.7 轻量编辑数据流（Story 9）

```
用户在 H5 页面内联编辑已激活实体：
  → 字段编辑/增删任务（物理删除）/阶段排序 → 落库
  → 状态变更（回退/暂停/恢复）→ 校验状态机 + reason → UPDATE status + status_change_log → 即时重算级联
  → 项目空间状态只读查看（managed/path，激活后不能改）
```
