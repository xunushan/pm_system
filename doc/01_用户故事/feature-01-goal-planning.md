# Feature: 目标规划与编辑 (Goal Planning & Editing)

## User Stories

### Story 1: 目标设立与拆解

**User Story**  
作为用户，我希望 PM 助手帮我系统规划一个长期目标，并拆解成可执行的方向、步骤和具体要做的事，全程在聊天中沟通确认，最后生成方案交由 H5 页面调整。

**When**  
用户主动向 PM 助手发送规划指令，如"帮我规划具身智能算法岗面试准备，计划3个月"。

**Dependencies**  
无。

**Workflow**  
```
用户：帮我规划一下具身智能算法岗面试准备，计划3个月

PM助手：这类面试通常需要这几个方向：
    1）深度学习基础
    2）面试准备
    3）具身智能专项
    你觉得还需要补充吗？比如 Agent 框架？

用户：加一个 Agent 框架

PM助手：好，4 个专题定了。计划多久？

用户：3 个月，7 月到 9 月，10 月投简历

PM助手：目标时间范围 7 月~9 月，计划开始日期 7 月 2 日，对吗？
    接下来逐个专题细化阶段和任务，先从深度学习基础开始：
    1）神经网络基础与 MLP
    2）优化器
    3）CNN
    4）Transformer
    阶段划得可以吗？

用户：可以

PM助手：那每个阶段的任务，我列一下：
    阶段1 MLP：前向推导、反向推导、手写实现、MNIST训练...
    任务要调整吗？

用户：加个 Dropout

PM助手：好，深度学习基础这个专题拆完了，可以吗？

用户：可以

PM助手：下一个专题：面试准备，继续？
    ...（逐专题讨论，每专题确认后进下一个）...

（所有专题讨论完后）

PM助手：4 个专题都拆完了：
    - 深度学习基础（6 阶段，24 任务）
    - 面试准备（3 阶段，12 任务）
    - 具身智能专项（5 阶段，20 任务）
    - Agent 框架（4 阶段，16 任务）
    （卡片：方案总览 + 确认方案按钮）

用户：点击"确认方案"

PM助手：方案已保存。后续调整请在 H5 页面完成：https://pm.example.com/plan/goal_001
```

**Acceptance Criteria**  
- 专题构成、目标时间范围、计划开始日期、逐专题的阶段与任务，均在聊天中与用户逐项确认
- 每个专题先确认阶段、再确认任务，用户确认后才进入下一个专题
- 全部专题确认完成后，推送 1 张总览卡片
- 总览卡片仅展示概览统计（专题数、阶段数、任务数），不展示任务级细节（细节在 H5 页面查看）
- 用户点击总览卡片"确认方案"后，方案正式保存
- 方案保存后返回 H5 页面链接，后续调整在页面进行
- 方案包含目标、专题、阶段、任务四层结构

**Boundary Scenarios**  
- 待定义：规划对话中途用户长时间不回复，系统应继续还是重新开始（原文未约定）
- 待定义：用户重复发送规划指令且已有进行中的规划对话时，系统是覆盖还是提示继续（原文未约定）
- 待定义：总览卡片发出后用户一直不点确认，卡片有效期及过期处理（原文未约定）

**Key Decisions**  
> - **规划环节职责收敛** - 只负责沟通与产出方案，产出后退场，不承担后续执行与调整。
> - **全程聊天沟通** - 专题、目标时间范围、逐专题阶段任务都在聊天中确认；卡片仅用于最终总览与确认。
> - **H5 页面承载调整** - 方案确认后返回 H5 链接，所有结构调整在页面完成。

### Story 9: 轻量编辑与状态回退

**User Story**  
作为用户，我希望在 H5 页面直接编辑已激活实体（改字段、改 deadline、增删任务、阶段排序、状态变更），并在标记错误时能回退。

**When**  
用户在 H5 页面内联编辑已激活实体。

**Dependencies**  
目标已存在且处于已激活状态。

**Workflow**  
- **场景1：改字段/deadline/executor** - 用户在 H5 页面改 deadline / executor / 名称 -> 系统保存，H5 同步显示
- **场景2：增删任务** - 用户在已激活阶段下新增/删除任务 -> 系统保存
- **场景3：阶段排序** - 用户拖拽阶段改顺序 -> 系统保存
- **场景4：状态变更 - 暂停** - 用户操作 进行中->已暂停，填 reason -> 系统更新状态并记日志（pause），不纳入计划/不占名额
- **场景5：状态变更 - 恢复** - 用户操作 已暂停->进行中 -> 系统更新状态并记日志（resume），重新纳入计划
- **场景6：状态变更 - 回退** - 用户操作 已完成->待执行，填 reason -> 系统更新状态并记日志（revert）+ 即时更新上级阶段/专题状态
- **场景7：查看项目空间状态** - 用户在专题详情查看托管/关联模式与路径（只读） -> 系统显示模式 + path。激活后不能改托管模式。

**Acceptance Criteria**  
- H5 页面支持编辑已激活实体的字段：名称、描述、deadline、executor
- 支持在已激活阶段下新增/删除任务
- 支持阶段拖拽排序
- executor 可修改（单选：人 / 智能体）
- 专题详情可只读查看项目空间模式（托管/关联）与路径；激活后不能修改托管模式
- 状态变更支持三种：暂停（必填原因）、恢复（不填原因）、回退（必填原因）
- 回退（已完成->待执行）后立即重新计算上级阶段/专题状态
- 所有状态变更记录原因与时间，可追溯审计
- 已暂停实体不纳入每日计划、不占用进行中名额

**Key Decisions**  
> - **任务删除后不可恢复** - 区别于模板的停用标记（可恢复），任务一旦删除即永久移除。
> - **项目空间状态只读查看** - 托管/关联模式与路径在激活时设置，H5 页面只读查看，激活后不可改。
> - **有限回退** - 仅支持 已完成->待执行（必填原因）、进行中↔已暂停。
> - **任务排序不在页面固定编辑** - 由每日计划动态决定；阶段排序保留。

---

## Non-Goals
- 规划态不定义 deadline 与 executor（由激活环节定义）
- 不实现任务排序编辑（由每日计划动态调度）
- 本期不实现卡顿监测

## Notes
- Story 1 与 Story 9 共同覆盖目标全生命周期：规划设立 -> 激活后编辑 / 状态回退
- 本 Feature 的产出（已确认方案）是 Feature-02（计划调度）的输入：Story 1 确认方案后，经 Story 2 激活阶段进入每日计划
