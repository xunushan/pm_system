# 文档治理范式

> 版本：v1.0
> 状态：accepted
> 更新：2026-07-10
> 定位：本文档是文档治理方法论的规范归宿（分层 / 模板 / 变更流程 / 渐进式披露），他处引用不复述
> 决策溯源：见 doc/07（本文档自身的治理决策待补 D{N}）

---

## 目的

本范式从 pm-system e2e 阶段复盘沉淀，目标是：

1. 解决"同一事实在多处重述、一处改多处漏"的文档漂移（e2e 期方案B 落地后 03/04/06 未同步即此症）。
2. 让子 agent 在开发/review 时只读必需内容，降低 token 消耗、防止跑偏。
3. 使需求/方案变更成为"发起-评估-执行-验证"的确定性闭环，而非靠记忆的零散改动。
4. 项目无关：本范式对任何 spec-first 项目都成立。

## 一、三层文档分层（铁律 / 关键决策 / 规范正文）

### 核心区分：稳定性 + 性质

| 层 | 性质 | 稳定性 | 内容 | 子 agent 读取 |
|---|---|---|---|---|
| **constitution（铁律）** | 行为准则 / 工程纪律 | 极稳、少动、全模块适用 | 测试全绿才提交、事务内禁 IO、先搜后建、commit 规范、状态变更写审计 | 全量必读（1 页） |
| **doc/07（关键决策 ADR）** | 面向项目/业务的技术选型 + 验证事实 | 易变、可废弃、强上下文 | DECISION（有权衡选型）+ FACT（spike/prototype 验证的外部行为，无权衡） | 仅相关 D{N}，按模块/Story 过滤 |
| **doc/0X 正文（spec/设计）** | 各域规范 | 中 | 用户故事、数据模型、架构、API、流程、UI 样式 | 按契约单指定文件，不读全量 |

### 铁律 vs 关键决策的本质差异

- **铁律 = 行为准则**：和具体技术选型无关，所有模块每次都适用、都遵守。"怎么做事"。例：测试全绿、事务内禁 IO、先搜后建。
- **关键决策 = 技术选型**：有上下文、有取舍、可废弃。"为什么选这个方案"。例：看板用 H5 不用多维表格（D9）、opencode 方案B、Service 不调 LLM（D3）。

> **纠偏**：`Service 不调 LLM` 是技术决策（D3），非铁律。它的约束力强≠它是铁律。铁律看"是否项目无关的行为准则"。强约束的技术选型仍归 doc/07，子 agent 按模块相关性读取。

### 三层的关系：指针 + 状态，不重复存储

- 关键决策可衍生铁律（如 D6 状态机扩充 -> 现为状态机铁律约束）。衍生后铁律在 constitution 自成体系、不依赖 ADR，仅标 `源自 D{N}` 单向追溯。
- 铁律也可过期（项目后续若让 Service 调轻量 LLM，对应铁律 superseded）。故 constitution 每条铁律标来源决策 + 可被标 superseded，状态机制与 ADR 统一。
- constitution 独立存铁律内容（1 页），不是 ADR 的指针视图--子 agent 读一次即够，不跳 ADR（避免多读文件）。

### 必备文档集

| 文档 | 定位 | 必要性 |
|---|---|---|
| 01 用户故事 | what/why（Spec Kit 的 spec） | 必备 |
| 02 数据模型 | 表 + 状态机 + 级联 | 必备 |
| 03 系统架构 | 分层 + 组件 + 集成约束（稳定/易变分离） | 必备 |
| 04 服务API | 端点契约 | 必备 |
| 06 操作流程与技术动作清单 | 介于用户故事与架构之间，拆「用户动作-系统行为」表，指导子 agent | 必备 |
| 07 关键决策（ADR） | 技术选型 + 验证事实 | 必备 |
| 09 UI 交互样式 | UI 卡片原型归宿，纯 UI | 必备 |
| 05 Skill 设计 | 有交互层 Skill 时需要 | 可选 |
| 08 教训 | 可并入 07 FACT | 可选 |

## 二、铁律与关键决策的分离规范

### 铁律（constitution）

- 位置：CLAUDE.md 的 constitution 区（或独立 `doc/00_constitution.md`，视项目惯例）。
- 性质：行为准则、极稳、全模块适用。
- 每条格式：
  ```
  - <铁律内容>（源自 D{N}）
  ```
- 状态：可标 superseded，但极少；废弃时标 `〔superseded, 见 D{N}〕`。
- 读取：子 agent 每次全量必读。

### 关键决策（doc/07 ADR 体系）

- 性质：技术选型（DECISION）+ 验证事实（FACT），易变、可废弃。
- 每条格式见 §三 模板。
- 读取：子 agent 只读与当前模块/Story 相关的 D{N}（按 Story/模块相关性过滤）。

### 衔接：决策固化/废弃

- DECISION 被长期接受、衍生出稳定行为准则 -> 可提炼一条铁律进 constitution，标 `源自 D{N}`。doc/07 该 D 状态不变（仍 accepted）。
- DECISION 被推翻 -> doc/07 标 `superseded by D{N}`，不得删（保审计链）。若其衍生的铁律随之废，constitution 那条标 superseded。
- FACT（外部验证事实）一般不"废弃"，只"被新 FACT 修正" -> 标 `corrected by D{N}`。

## 三、各文档模板定义

### 通用 front matter（所有文档统一）

```markdown
> 版本：vX.Y
> 状态：accepted | deprecated | superseded
> 更新：YYYY-MM-DD
> 定位：本文档是 ___ 的规范归宿（单一写入，他处引用不复述）
> 决策溯源：见 doc/07（D{N} 起影响本文档）
```

**禁止**：front matter 塞 changelog 列表（如"变更 1…10"）。版本演化归 doc/07 + git，正文写当前真相。历史变更 = 决策，归 doc/07。

### doc/01 用户故事（Spec Kit 的 spec，what/why）

```markdown
[通用 front matter]
# 用户故事
## Story 清单
### Story N：<名称>
- 用户视角：作为<角色>，我想<动作>，以便<价值>
- 前置条件：
- 主流程：编号步骤
- 替代流程 / 异常：
- 验收标准：Given / When / Then
```

### doc/02 数据模型

```markdown
[通用 front matter]
# 数据模型
## 表清单（index：每表一行「表名 - 用途 - 链接」）
## 表定义（每表：字段 / 类型 / 约束 / 索引 / 级联）
## 状态机（阶段 / 任务流转，独立成节，多 Story 复用）
## 级联规则（独立成节）
```

### doc/03 系统架构（稳定 / 易变分离）

```markdown
[通用 front matter]
# 系统架构
## 行为准则：见 constitution（此处不重复铁律内容）
## 架构概述（分层 + 组件）
## 关键技术决策：见 doc/07（此处只索引相关 D{N}，不复述）
## 集成约束（易变，spike 驱动，行为指向归宿，不复述）
  ### 飞书（约束，非行为复述；行为见 doc/09 / doc/07 FACT）
  ### opencode
```

### doc/04 服务API

```markdown
[通用 front matter]
# 服务API
## 端点清单（index：方法 路径 - 用途 - 链接）
## 端点定义（每域：请求 / 响应 / 错误 / 回调）
```

### doc/06 操作流程与技术动作清单（指导子 agent 的关键文档）

```markdown
[通用 front matter]
# 操作流程与技术动作清单
## 一、操作流程（每 Story）
### Story N：<名称>
| 步骤 | 用户动作 | 系统行为 | 依据决策 |
（步骤后跟「要点」归纳约束）

## 二、技术动作清单
- 表1 用户指令（/pm 类）
- 表2 卡片按钮点击（入口 B）
- 表3 H5 页面操作（入口 C）
- 表4 定时任务

## 三、关键技术决策索引（指向 doc/07 D{N}，不复述）
```

> 「依据决策」列把每步关联到 D{N}，变更时反查影响范围。

### doc/07 关键决策（ADR 体系）

```markdown
[通用 front matter]
# 关键决策
## DN. <标题>
- 类型：DECISION | FACT
- 状态：accepted | superseded by D{N} | corrected by D{N}
- 上下文：为什么需要这个决策（1-2 句）
- 决策：选了什么（仅 DECISION 有；FACT 无权衡，跳过此节）
- 证据：（可选，spike / e2e / PR 链接；无则省，不强求）
```

> **不加 Updates（影响文档）字段**：决策文档只管"决策是什么"，不维护"影响哪些文档"的台账（那是动态的，常驻字段会双写且迟早漏、漏了比没有更危险）。决策落地后正文跟着清，靠 §四变更流程的**一次性影响检测**（grep 关键词遍历 doc/），不靠字段。
> **证据可选**：历史决策可能无对应 PR/spike（讨论共识），强制填会造假。有则填，无则省。
> **superseded 的旧 D 压缩成墓碑**：`D1. <标题> 〔superseded by D9〕` + 一句推翻原因，不保留旧决策完整正文（正文在新 D 里）。同一主题只留一份完整内容。
> **决策文档边界**：只存决策本身（类型/状态/上下文/决策/+可选证据）。背景归 doc/01、原则归 constitution、过程对比/影响纵览归 git/archive、未决项开 issue--这些不进 doc/07。

### doc/09 UI 交互样式（纯 UI 原型归宿）

```markdown
[通用 front matter]
> 定位：本文档是 UI 卡片交互样式的规范归宿（schema / 布局 / 按钮 / 视觉）
> 非 UI 的外部 API 行为验证见 doc/07（FACT 类），此处不收
# 卡片交互样式
## 通用规则（schema 2.0 结构、按钮两类、回调后必须刷新）
## 每 Story 卡片样式（初始态 / 确认态 / 刷新态 JSON + 说明）
## 官方文档链接
```

> **关键约束**：doc/09 只收 UI。非 UI 验证事实（飞书 API 行为、opencode session 模型、级联事务行为）归 doc/07 FACT。这避免"UI 文档越界兜底集成验证"。

### 模板存放与复用

模板集中存 `doc/_templates/`，新项目 `cp` 后填。front matter 统一，禁各文档自创格式。

## 四、需求/方案变更流程（发起-追问-影响检测-执行-验证）

### 闭环

```
①发起  ②追问澄清  ③影响范围检测  ④记录决策（立 D{N}）  ⑤执行变更  ⑥验证一致
```

### ①发起

任何变化（e2e 发现 / 用户改主意 / spike 验证 / bug）先建变更源：GitHub issue（标 `type:change`）或 doc/07 一条 `proposed` D。必含：变更内容 + 变更理由（为什么不能维持现状）。

### ②追问澄清

发起后先澄清模糊点（grilling 一问一答原则）：变更边界、是否推翻既有决策、是否有替代方案。未澄清不进影响检测。

### ③影响范围检测（关键，防漏）

用清单扫四类影响，不靠记忆：

| 影响类型 | 评估方法 |
|---|---|
| 文档 | grep 变更关键词遍历 doc/，列命中文件 + 是否需改 |
| 关键决策 | 是否推翻既有 D？标 superseded |
| 代码 | grep app/ 列受影响模块 + 是否需改 |
| 测试 | 受影响功能的测试是否补/改 |

结论写进变更源 issue（受影响文档/代码清单），形成"变更 TODO 清单"。

### ④记录决策

确认要变，在 doc/07 立 D{N}（类型/状态/上下文/决策）。受影响文档清单存 issue（③），不进 D。

> **顺序铁律**：先立 D 再改正文。不能"先改正文后补决策"--后者正是漂移成因。受影响文档清单存 issue，按清单逐个改。

### ⑤执行变更

按 issue 清单逐个改文档 + 改代码。每改完一项清单打勾。

### ⑥验证一致

变更后跑一致性扫描（§五）确认文档不漂移。

## 五、变更影响评估与承载形式

变更对三层承载物的影响分工：

| 承载物 | 变更影响 | 评估结论承载形式 |
|---|---|---|
| **progress.md** | 变更是否改变当前焦点/下一步 | 「下一步」直接改（<40 行，可重写，不存详情） |
| **issues** | 变更拆出的开发任务/问题 | 开 issue（标 type/story/优先级，自包含含复现+验收） |
| **代码** | 受影响模块 + 回归测试 | 开 issue（`Fixes/Refactors`，含受影响文件清单+验收） |
| **issue** | 文档一致性 TODO + 变更拆解工作 | issue body 存受影响文档/代码清单 |

### 一致性验证（可执行检查）

```
scripts/doc-consistency-check.sh
  - 扫开放 issue 的受影响文档清单
  - 对每个受影响文档，grep 相关决策关键词，确认文档内容与决策一致
  - 输出：✓ 一致 / ✗ doc/03 §8.x 与 D27 不一致（待修）
```

把"文档一致性"从靠记忆/review 变成可执行检查，类似 `make lint` 之于代码。变更后跑一次，漂移立即暴露。

## 六、渐进式披露文档结构

### 原则

文档拆分的真正目的不是"让子 agent 直接读拆分文件"，是"让主 agent 能精准抽取契约单"。拆分粒度判据：**能否独立成一张任务契约单的输入**。

### 结构

每文档有 index（目录 + 一行摘要），子文件按关注点拆，CLAUDE.md Story 表指向精确路径。

例（doc/02）：
```
doc/02_数据模型/
├── index.md          # 表清单：每表一行「表名 - 用途 - 链接」+ 状态机/级联入口
├── tables/
│   ├── goal.md       # 单表：字段/约束/索引
│   └── ...
├── state-machine.md  # 独立成节，多 Story 复用
└── cascade-rules.md
```

### 三场景读取模型

| 场景 | 加载内容 | 省了什么 |
|---|---|---|
| 主 agent 开新 session | 瘦 CLAUDE.md（铁律+指针）+ progress.md（<40 行）+ MEMORY.md | 不全量加载长文档；按指针读 |
| 子 agent 实现代码 | 任务契约单（per-Story）+ 目标文件 + 铁律速查单（1 页） | 不读全 CLAUDE.md、不读全 doc/0X |
| code-reviewer | PR diff + 铁律速查单（1 页） | 不读全 CLAUDE.md |

### 任务契约单（主 agent 派发子 agent 时抽取）

```
Story: N <名称>
模型: <sonnet|haiku>（含业务逻辑用 sonnet，机械用 haiku）
铁律: <§相关条目>
表: doc/02/tables/<表>.md
端点: doc/04/<域>.md
复用件: app/core/<件>.py
范本: app/models/goal.py
测试 case: <输入/期望>
验收: make test + make lint 全绿，PR title 含 StoryN
```

子 agent 只读：契约单 + 表文件 + 复用件 + model 范本 + 测试骨架。抽取成本由主 agent 在规划期付一次。

## 附：范式总结（项目无关）

1. **三层分离**：铁律（行为准则，constitution）vs 关键决策（技术选型 + 事实，doc/07 ADR）vs 规范正文（doc/0X）。稳定性递减。
2. **单一写入**：每事实一处规范归宿，他处引用不复述。维护成本 ∝ 重复度，非文档数。
3. **决策联动**：决策文档只存决策本身（不背 Updates 影响台账）。决策落地一致性靠 §四变更流程的一次性影响检测，不靠常驻字段。
4. **变更闭环**：发起-追问-影响检测-记录-执行-验证。影响检测用清单不靠记忆，验证用脚本。
5. **渐进式披露**：index + 关注点子文件 + 契约单。子 agent 读契约单不读全文档。
