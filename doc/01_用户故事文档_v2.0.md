# 一、名词解释

| 系统概念 | 用户可见表述 | 说明 |
|----------|-------------|------|
| 目标 | 目标 | 用户要达成的最终状态 |
| 专题 | 方向 / 板块 | 目标下的独立推进板块（无序并列） |
| 阶段 | 步骤 | 专题内的可验收推进节点，按 roadmap 顺序推进（强约束） |
| 任务 | 具体要做的事 | 阶段内的可执行事项 |
| 工作空间 | 项目空间 | 与专题绑定的物理目录 |
| 项目空间托管模式 | （内部概念） | managed=1 系统托管（激活初始化）；managed=0 关联已有路径（跳过初始化，不创建任何文件） |
| H5 页面 | 页面 | 规划确认后的调整入口，承载已激活实体的 CRUD、deadline、状态变更 |
| 执行前工作 | 执行前工作 | 任务开始前自动准备的辅助工作（只服务人执行任务） |
| 执行后工作 | 执行后工作 | 任务完成后自动整理的收尾工作（只服务人执行任务） |

# 二、用户故事

## Story 1：目标设立与拆解（原 Story 0+1）

### 角色
作为用户，我想让 PM 助手帮我系统规划一个长期目标，并拆解成可执行的方向、步骤和具体要做的事。全程在聊天中沟通确认，最后生成方案给 H5 页面调整。

### 触发条件
用户主动向 PM 助手发送规划指令，如"帮我规划具身智能算法岗面试准备，计划3个月"。

### 前置条件
无。

### 主流程

```
你：帮我规划一下具身智能算法岗面试准备，计划3个月

PM助手：这类面试通常需要这几个方向：
    1）深度学习基础
    2）面试准备
    3）具身智能专项
    你觉得还需要补充吗？比如 Agent 框架？

你：加一个 Agent 框架

PM助手：好，4 个专题定了。计划多久？

你：3 个月，7 月到 9 月，10 月投简历

PM助手：目标时间范围 7 月~9 月，计划开始日期 7 月 2 日，对吗？
    接下来逐个专题细化阶段和任务，先从深度学习基础开始：
    1）神经网络基础与 MLP
    2）优化器
    3）CNN
    4）Transformer
    阶段划得可以吗？

你：可以

PM助手：那每个阶段的任务，我列一下：
    阶段1 MLP：前向推导、反向推导、手写实现、MNIST训练...
    任务要调整吗？

你：加个 Dropout

PM助手：好，深度学习基础这个专题拆完了，可以吗？

你：可以

PM助手：下一个专题：面试准备，继续？
    ...（逐专题讨论，每专题确认后进下一个）...

（所有专题讨论完后）

PM助手：4 个专题都拆完了：
    - 深度学习基础（6 阶段，24 任务）
    - 面试准备（3 阶段，12 任务）
    - 具身智能专项（5 阶段，20 任务）
    - Agent 框架（4 阶段，16 任务）
    （卡片：方案总览 + 确认方案按钮）

你：点击"确认方案"

PM助手：方案已保存。后续调整请在 H5 页面完成：https://pm.example.com/plan/goal_001
```

### 设计要点
- **pm-plan 职责收敛**：只沟通+产出规划，产出后退场。
- **全程聊天沟通**：专题构成、目标时间范围、scheduled_start_date、逐专题的阶段任务，都在聊天讨论确认。
- **逐专题确认**：每专题先确认阶段、再确认任务，用户确认后进下一个。逐专题生成的内容追加到同一个 draft。
- **总览卡片触发写库**：所有专题讨论完，发总览卡片（只展示概览，不展示全部细节）+ 确认按钮。
- **drafts 表用于确认前存储**：pm-plan 生成规划时调 Service 写入 drafts（content 存完整规划 JSON，含目标+专题+阶段+任务）。确认按钮回调只传 draft_id（规避飞书回调约 30KB 数据量限制，规划数据可达几十 KB）。Service 收到回调，用 draft_id 读 drafts，写正式表，删 drafts。drafts 纯存储，不同步展示（不进 H5 页面/多维表格）。
- **确认后给 H5 链接**：落库后返回 H5 页面链接，后续调整在页面。
- **无 is_draft 机制**：规划数据存 drafts，不存正式表（不污染正式表）。
- **规划态产出**：目标（名称+时间范围+scheduled_start_date+描述）、专题（名称+类型+排序）、阶段（名称+排序，无 deadline）、任务（名称，无 executor）。

### CRUD 场景（确认后，在 H5 页面）

用户在 H5 页面可做的调整（详见 Story 9）：
- 目标层：改目标名称、描述、时间范围
- 专题层：新增/删除专题、改专题名称/描述/类型
- 阶段层：新增/删除阶段、改阶段名称/描述、阶段排序
- 任务层：新增/删除任务、改任务名称/描述
- deadline、scheduled_start_date、executor、状态变更：激活后相关操作（Story 9）

### 验收标准
- 全程聊天沟通确认专题构成、目标时间范围、scheduled_start_date、逐专题阶段任务
- 逐专题确认后进下一个，全部完成后发总览卡片
- 总览卡片只展示概览（专题数、阶段数、任务数），不展示全部细节（细节在 H5 页面）
- 用户点总览卡片"确认方案"，回调只传 draft_id
- Service 用 draft_id 读 drafts 写正式表，删 drafts
- 落库后返回 H5 页面链接，后续调整在页面
- 方案包含目标、专题、阶段、任务四层结构
- 规划态不定 deadline、executor

### 关键决策

> **pm-plan 职责收敛** — pm-plan 只沟通+产出规划，产出后退场。

> **全程聊天沟通** — 专题、目标时间范围、逐专题阶段任务都在聊天确认。卡片只在最后总览+确认用。

> **drafts 表用于确认前存储** — pm-plan 生成规划写入 drafts（纯存储，不同步展示）。确认按钮回调只传 draft_id，规避飞书回调约 30KB 数据量限制。

> **H5 页面承载调整** — 确认后给 H5 链接，所有调整在页面。

> **无 is_draft 机制** — 规划数据存 drafts，不存正式表。

## Story 2：首次纳入调度（含项目空间设置）

### 角色
作为用户，我想激活规划好的目标中的步骤，开始实际推进。用 patch 卡片形式：选专题+设项目空间 → 填 deadline → 确认激活。原 Story 8（项目空间关联）并入本 Story。

### 触发条件
用户发送"开始调度"或 H5 页面点"激活"。

### 前置条件
目标已存在且状态为"未开始"或"进行中"。

### 主流程

```
你：/pm 开始调度

PM助手：（卡片 A：选专题 + 设项目空间）
    多选要推进的专题（专题无序，并列）：
    ☑ 深度学习基础    [托管 ▼]   ← 默认托管
    ☐ 面试准备        [托管 ▼]
    ☐ 具身智能专项    [关联 ▼] path: /Users/me/projects/embodied  ← 已关联
    ☐ Agent 框架     [托管 ▼]
    [下一步]

你：勾选"深度学习基础"和"面试准备"，把"面试准备"改成关联 /Users/me/interview
   点"下一步"

PM助手：（卡片 A patch 为卡片 B：填 deadline + 确认项目空间）
    将激活：
    · 神经网络基础与 MLP（深度学习基础，托管）   deadline：[date_picker]
    · 算法基础（面试准备，关联 /Users/me/interview） deadline：[date_picker]
    [确认调度]

你：填各 deadline，点"确认调度"

PM助手：调度已确认，2 个阶段已激活。
    · 神经网络基础与 MLP：项目空间已就绪（系统托管，已初始化）
    · 算法基础：项目空间已就绪（关联已有，跳过初始化）
    从今天起纳入每日计划。
```

### 设计要点
- **patch 卡片形式**：选专题+设项目空间（卡片 A）→ patch 为填 deadline（卡片 B）。
- **多选专题**：一次激活多个专题，每个专题激活其第1个未开始阶段。
- **专题无序，并列多选**。
- **阶段自动锁定**：选了专题，系统自动锁定该专题 sort_order 最小的未开始阶段（强约束，用户不选阶段）。
- **项目空间设置在卡片 A**：每个专题显示 managed 状态（默认托管，可改关联+填 path）。已关联的显示当前 path。
- **managed=0（关联）**：校验 path 存在性，不创建任何文件（包括规范文件），用户自己保证目录有规范文件。
- **deadline 在卡片 B 填**：每个阶段一个 deadline，必填。
- **不碰 scheduled_start_date**：它在 Story 1 已确认。
- **首次与追加统一**：中途追加激活走同一流程。
- **激活后不能改 managed**（避免破坏已初始化目录）。
- **全局进行中 ≤3 校验**（已暂停不占名额）：多选专题数+现有进行中 ≤3。

### 验收标准
- patch 卡片形式（选专题+设项目空间→填 deadline）
- 多选专题，每个专题激活其第1个未开始阶段
- 阶段自动锁定（强约束）
- 卡片 A 显示各专题 managed 状态，可改关联+填 path
- managed=0 校验 path，不创建任何文件
- managed=1 系统托管，初始化（mkdir+git init+骨架含规范文件）
- deadline 必填
- 全局进行中 ≤3
- 激活时即时级联 themes/goals 到进行中
- 记录 phases.activated_at（每日计划以此为准）
- 首次与追加统一流程
- 激活后不能改 managed

### 关键决策

> **patch 卡片形式** — 选专题+设项目空间→patch 填 deadline，避开飞书卡片级联限制。

> **多选专题** — 跨专题并行推进是实际需求，一次激活多个专题。

> **项目空间设置在激活时** — managed/path 在卡片 A 设置（激活前最后一刻），原 Story 8 并入。H5 页面专题详情可只读查看 managed/path。

> **阶段强约束** — 专题内阶段按 sort_order 顺序激活，自动锁定第1个未开始阶段。

> **scheduled_start_date 在 Story 1 确认** — Story 2 不碰，激活后用 phases.activated_at 推送。

> **激活后不能改 managed** — 避免破坏已初始化目录。

## Story 3：当日计划推送

### 角色
作为用户，我每天收到 PM 助手推送的当日任务清单。今日计划由 LLM 综合昨日完成+进度+deadline 决策，卡片带候选供勾选。

### 触发条件
- 定时：8:30 自动
- 主动：用户发送"今日计划"
- 巡检：10:00 未确认计划时提醒

### 前置条件
存在已激活阶段（phases.activated_at 有值）。

### 主流程

```
（8:30 定时触发）

PM助手：📋 今日计划（2026-07-02）
    昨日完成：反向传播推导 ✅、手写 MLP PyTorch ✅

    今日候选任务（勾选今天要做的，3-5 个）：
    ☑ 优化器专题（神经网络基础与 MLP(7/15)）[人]
    ☑ 手写 MLP PyTorch（神经网络基础与 MLP(7/15)）[人]
    ☐ MNIST 端到端训练（神经网络基础与 MLP(7/15)）[人]
    ☐ 数组专题 2 题（算法基础(7/20)）[人]
    ☐ 项目复盘提纲（面试准备(7/25)）[智能体]

    执行前工作（勾选要执行的，可取消）：
    ☑ 搜集优化器相关资料
    ☑ 准备 PyTorch 开发环境
    ☐ 获取今日 LeetCode 题目
    [确认今日计划]

你：勾选 3 个任务 + 取消"获取今日 LeetCode 题目"，点"确认今日计划"

PM助手：今日计划已生效，执行前工作进行中。

（5 分钟后）
PM助手：执行前工作已完成 ✅
    · 优化器资料已搜集
    · PyTorch 环境已就绪
    你可以开始学习了。
```

### 设计要点
- **今日计划走卡片**：pm-daily LLM 决策 + 卡片勾选。
- **LLM 综合决策**：综合昨日完成+进度+deadline+依赖，带出 5-8 个候选。
- **executor 自动推断**：根据专题 type 推断（learning/research/source→人；dev/survey→智能体），卡片只读展示，要改走 H5 页面。
- **候选任务展示形态**：`任务名（阶段名(deadline)）[executor]`，紧凑。
- **任务池过滤**：已激活阶段（activated_at 有值）下的待执行任务 + 排除已暂停。
- **前置按今日整体生成**：pm-daily 调 pm-subtask 生成"今日整体前置"（不按单个任务），与任务解耦。
- **前置和任务两组独立勾选**：用户分别勾选任务和前置，互不影响。取消任务不影响前置（用户自己判断）。
- **前置只对人执行任务**：只对 executor=人 的任务生成前置，智能体任务不生成前置。
- **前置可取消**：用户可勾选/取消前置，确认后异步执行（智能体执行）。
- **无草案态**：确认前数据在卡片，确认时直接写库。
- **不确认只 10:00 提醒**，不自动执行。
- **完成走 4B**。

### 验收标准
- 今日候选任务由 LLM 综合决策，带出 5-8 个供勾选
- executor 根据专题 type 推断（规则在 pm-daily Skill），卡片只读展示
- 候选任务展示：任务名（阶段名(deadline)）[executor]
- 任务池过滤已激活阶段 + 排除已暂停
- 前置按今日整体生成，与任务解耦，两组独立勾选
- 前置只对 executor=人 的任务生成，可取消
- 用户只能勾选/取消候选任务，不能加新任务
- 卡片勾选，用户选 3-5 个任务确认
- 无草案态
- 不确认只 10:00 提醒，不自动执行
- 完成走 Story 4B，卡片不承担完成确认

### 关键决策

> **LLM 决策今日计划** — pm-daily 综合昨日完成+进度+deadline+依赖决策。

> **executor 自动推断** — 根据专题 type 推断，卡片只读，要改走页面。

> **前置按今日整体生成 + 与任务解耦** — 前置不按单个任务，两组独立勾选，取消任务不影响前置。前置由 pm-subtask（LLM）生成，Service 不调 LLM。

> **前置只对人执行任务** — 智能体任务不生成前置。

## Story 4：工作空间任务执行

### 4A：执行主体为智能体（代码开发、文档生成）

```
（承接 Story 3 确认今日计划，异步触发）

系统：判断勾选任务是否有 executor=智能体？
  - 是 → 查工作空间路径，启动 opencode serve（首次），下发任务
  - 设置 Redis 超时 2h

OpenCode：执行，按项目空间规范（claude.md 类）写产出文件

（产出完成，OpenCode 回调）

系统：记录产出 → DEL 超时 → 发验收卡片 + 发送产出文件到飞书：
    📋 任务完成确认
    任务：设计消息 Schema
    产出文件：
    · 消息Schema-v1.md
    · schema-test.py
    （按钮：验收通过 / 需要修改）
    + 文件消息（用户在飞书直接看）

你：看文件，点"验收通过"

系统：UPDATE tasks 已完成 + 即时级联 + 发阶段完成事件 → Supervisor 衔接

（若 3 次重试不通过）
系统：opencode serve 退出 + 飞书通知 + 工作空间路径 + 建议手动启动

你：手动启动 opencode，给详细指令驱动产出，把关

你：/pm 确认完成 设计消息 Schema

系统：标记完成 + 即时级联 + 重新启动 opencode serve（不同端口）接管后续智能体任务
```

### 4A 设计要点
- **智能体在 Story 3 确认后唤起**（不是 Story 2 激活时）。
- **产出统一为文件**（测试结果也写文件），规范在项目空间骨架文件（claude.md 类）。
- **验收用文件名体现产出**，验收卡片 + 发送产出文件到飞书。
- **feedback 卡片输入框直接提交**，重试上限 3 次。
- **3 次不通过不改状态**，系统进程退出，用户手动接管。
- **用户"/pm 确认完成"后系统重新启动 opencode serve（不同端口）接管后续智能体任务**。
- **即时级联**，不生成前置/后置。

### 4A 验收标准
- 智能体任务在 Story 3 确认后异步唤起，阶段级常驻，动态端口
- 产出统一为文件，按项目空间规范写
- 验收卡片 + 发送产出文件到飞书（用文件名体现产出）
- 验收通过即时级联 + 发阶段完成事件
- 需要修改 feedback 卡片输入框直接提交，重试上限 3 次
- 3 次不通过：系统进程退出 + 通知 + 工作空间路径 + 建议手动启动
- 用户手动启动处理，"/pm 确认完成"后系统重新启动 opencode serve（不同端口）接管后续智能体任务
- 2 小时超时 Redis 兜底告警
- 智能体任务不生成前置/后置

### 4B：执行主体为人（学习、刷题）


### 4B 设计要点
- **完成入口走指令**："/pm 完成 [任务]"。
- **完成和后置脱钩**：pm-subtask 调 Service 标记完成（即时级联）后，再生成后置清单。任务在标记完成时就已完成，不等后置确认。
- **后置生成**：pm-subtask（type=后置），只对人执行任务。基于模板+工作空间文件快照，LLM 生成。
- **后置可全取消**：后置是可选收尾，用户可全取消（点"不需要后置"），任务仍是已完成。
- **后置勾选**：默认全选，用户取消不想要的，不能新增。
- **后置确认后立即异步执行**（智能体执行，opencode run）。
- **后置失败非阻塞**。
- **不保留"需人工介入"**。

### 4B 验收标准
- 人做完任务后通过"/pm 完成 [任务]"指令触发
- pm-subtask 标记完成（即时级联）+ 生成后置清单
- 任务在标记完成时就已完成，不等后置确认（脱钩）

- 不保留"需人工介入"

> **即时级联** — 任务完成即时向上级联，不再依赖日终总结。

> **智能体任务不生成前置/后置** — 前置/后置只服务人执行任务，帮人完成准备和收尾。

> **后置和完成脱钩** — 完成时先标记（即时级联），后置是可选收尾，用户可全取消。

> **前置/后置由 LLM 生成** — pm-subtask 生成，Service 不调 LLM。

> **去掉"需人工介入"终态（4A）** — 3 次不通过不改状态，用户介入后"/pm 确认完成"，系统重新接管后续智能体任务。

> **两个完成指令** — 4B 用"/pm 完成"（带后置生成），4A 人工确认用"/pm 确认完成"（不带后置）。

> **验收发文件** — 验收卡片 + 发送产出文件到飞书，用文件名体现产出。
